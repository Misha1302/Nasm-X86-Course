# NASM x86 для олимпиадников на C++ — учебник-конспект

Это не справочник команд и не сухой силлабус. Это учебник, по которому можно реально идти день за днём: читать, рисовать схемы, писать маленькие фрагменты, ошибаться и быстро чинить ошибки.

Главная идея курса:

> Ты уже умеешь C++. Теперь учимся видеть, как твой код выглядит глазами процессора IA-32.

Формат курса:

- **NASM x86 / IA-32**, без задач на x64;
- разработка в **SASM**, но обязательно учимся собирать из терминала Linux;
- сначала закрываем домашки **01-4, 01-8, 01-14, 01-15, 01-16**;
- потом добираем экзаменационные темы: флаги, стек, CDECL, адресация, структуры, безопасность, x87.

Как читать:

1. Сначала пойми картинку в голове.
2. Потом разбери пример руками.
3. Потом закрой конспект и попробуй объяснить вслух.
4. Если не можешь нарисовать регистры, память, стек или флаги после инструкции — тему ещё рано считать понятой.

x64 существует, но пока не трогаем глубоко. В этом курсе задачи и базовая модель идут через IA-32: `eax`, `ebx`, `ecx`, `edx`, `esp`, `ebp`, стековые аргументы и CDECL.

## Проверяемые примеры

Ключевые фрагменты вынесены в `examples/*.asm`. Их можно собрать так:

```bash
for file in examples/*.asm; do
    name="$(basename "$file" .asm)"
    nasm -f elf32 "$file" -o "/tmp/$name.o"
    gcc -m32 -no-pie "/tmp/$name.o" -o "/tmp/$name"
    "/tmp/$name"
done
```

Это не заменяет чтение курса, но защищает от ситуации “в тексте красивый пример, а на деле он не собирается”.

## Базовая команда сборки

Для учебных примеров курса используем:

```bash
nasm -f elf32 main.asm -o main.o
gcc -m32 -no-pie main.o -o main
./main
```

`-no-pie` нужен потому, что первые NASM-примеры используют простые абсолютные адреса меток вроде `x`, `fmtOut`, `.table`. PIE/GOT/PLT — отдельная тема, её лучше не смешивать с первыми задачами.

## Перед стартом. Где ассемблер реально нужен

Прежде чем уходить в `eax`, стек и флаги, полезно честно ответить на простой вопрос: **зачем вообще кому-то ассемблер в 2026 году?**

Короткий ответ: почти никто не пишет на нём большие прикладные программы целиком. Но ассемблер всё ещё нужен там, где обычный язык слишком далеко от железа.

### Три главные причины

| Причина | Что это значит простыми словами | Пример задачи |
|---|---|---|
| Привязка к конкретной архитектуре | Нужно использовать конкретные инструкции конкретного процессора | загрузчик, ядро ОС, SIMD-инструкция, системный код |
| Производительность | Нужно ускорить очень горячий участок, который выполняется миллионы раз | кодек, криптография, обработка строк/памяти, математика |
| Полный контроль | Нужно управлять регистрами, стеком, адресами, ABI и машинным кодом напрямую | JIT, рантайм языка, reverse engineering, низкоуровневая отладка |

То есть ассемблер обычно появляется не потому, что “так круче”, а потому что **на этом уровне нужна точность, скорость или контроль**.

### Где он встречается конкретнее

Вот нормальный список областей, где ассемблер действительно может быть уместен:

- **Операционные системы, драйверы, загрузчики.** Там нужно работать с режимами процессора, прерываниями, регистрами, таблицами, соглашениями вызова и железом.
- **Компиляторы, интерпретаторы, JIT.** В какой-то момент нужно сгенерировать или понять машинный код.
- **Runtime-библиотеки языков.** Например, низкоуровневые части, связанные со стартом программы, переключением контекстов, атомарными операциями, памятью, строками, вызовами и ABI.
- **Высоконагруженные библиотеки.** Кодеки, криптография, обработка изображений, математика, `memcpy`/`memmove`/`strlen`-подобные функции.
- **HFT и low-latency системы.** Не потому что “всё пишут на asm”, а потому что иногда важны наносекунды и предсказуемость.
- **Reverse engineering и security.** Чтобы понять, что реально выполняется, нужно читать машинный код и дизассемблирование.
- **Embedded и микроконтроллеры.** Там может быть мало памяти, особое железо и нестандартные ограничения.

### Важная честность

Плохая формулировка:

> “Ассемблер нужен для максимальной скорости, значит всё надо писать на asm.”

Нормальная формулировка:

> “Ассемблер используют точечно: в местах, где нужна конкретная инструкция, конкретный ABI, очень низкий overhead или полный контроль над машиной.”

Современный компилятор часто пишет отличный машинный код. Поэтому хороший low-level программист не бросается переписывать всё на asm. Он сначала измеряет, смотрит дизассемблирование, понимает bottleneck, а потом уже решает, нужен ли ручной код.

### Примеры, которые можно упоминать аккуратно

Ниже — полезные примеры для рассказа ученикам. Я специально помечаю их как **требующие проверки перед публичным докладом**, потому что точные формулировки, размеры и состав кода могут меняться.

| Пример | Как формулировать аккуратно |
|---|---|
| FFmpeg | “В FFmpeg и похожих медиабиблиотеках в горячих местах часто используют SIMD и архитектурно-зависимые оптимизации, включая hand-written assembly.” Не надо говорить, что “весь FFmpeg написан на asm”. |
| KolibriOS | “Есть маленькие ОС, где ассемблер используется очень широко; часто в пример приводят KolibriOS/FASM.” Перед публичным использованием лучше проверить актуальный размер, состав и формулировку “полностью/почти полностью”. |
| .NET Runtime | “В рантаймах языков бывают платформенно-зависимые низкоуровневые куски: startup, GC helpers, interop, atomics, memory/string routines и т.п.” Не стоит без проверки утверждать, что конкретно “строки написаны на asm” в текущей версии. |
| libc / стандартные библиотеки C | “В реализациях libc горячие функции работы с памятью и строками могут иметь архитектурно-оптимизированные версии.” Конкретный проект и конкретные функции лучше проверять отдельно. |
| “Ребята из ядра / Синтакора / свой stdlib C” | Это оставить как заметку “проверить”. Нужно уточнить название проекта, что именно они разрабатывают и где там реально используются asm-вставки. |

Хороший стиль для ученика: **использовать примеры как направление мысли, но не превращать непроверенную байку в факт**.

### Почему тогда мы учим именно NASM x86

Потому что это удобная учебная площадка:

- регистров немного, их можно держать в голове;
- стековые вызовы функций хорошо видны;
- `ebp`, `esp`, `[ebp+8]`, `[ebp-4]` прямо показывают, как устроены аргументы и локальные переменные;
- задачи курса ВШЭ используют `nasm-x86`;
- на IA-32 проще почувствовать связь C/C++ и машинного кода.

x64 тоже важен, но его ABI и модель вызовов другие. Мы до него дойдём только обзорно, чтобы не смешивать две системы в голове.

## Введение для новичков. Самые нужные инструкции NASM x86

Перед первым днём полезно иметь маленькую карту местности. Не надо учить эту таблицу наизусть за один вечер. Смысл другой: когда ты видишь незнакомый фрагмент, быстро находишь команду и вспоминаешь, что она делает.

В NASM почти всегда порядок такой:

```asm
instruction destination, source
```

То есть:

```asm
mov eax, ebx    ; eax = ebx
add eax, 5      ; eax = eax + 5
```

И ещё одно правило, которое спасёт много нервов:

```asm
x       ; адрес метки x
[x]     ; значение в памяти по адресу x
```

### 1. Пересылка данных

| Команда | Пример | Что делает | Как думать на C++ |
|---|---|---|---|
| `mov` | `mov eax, ebx` | копирует значение | `eax = ebx` |
| `mov` | `mov eax, [x]` | читает значение из памяти | `eax = x` |
| `mov` | `mov [x], eax` | записывает значение в память | `x = eax` |
| `movsx` | `movsx eax, byte [x]` | расширяет маленькое signed-значение | `int eax = (signed char)x` |
| `movzx` | `movzx eax, byte [x]` | расширяет маленькое unsigned-значение нулями | `int eax = (unsigned char)x` |
| `lea` | `lea eax, [ebx+4*ecx]` | считает адрес/формулу, не читает память | `eax = ebx + 4*ecx` |
| `push` | `push eax` | кладёт значение на стек | `stack.push(eax)` |
| `pop` | `pop eax` | снимает значение со стека | `eax = stack.pop()` |

Главная ловушка здесь — `lea`. Название похоже на “load”, но память она не читает. Она просто считает выражение в квадратных скобках.

```asm
mov eax, [ebx+4*ecx]   ; прочитать a[i]
lea eax, [ebx+4*ecx]   ; посчитать адрес &a[i]
```

### 2. Целочисленная арифметика

| Команда | Пример | Что делает | Комментарий |
|---|---|---|---|
| `add` | `add eax, ebx` | `eax += ebx` | signed/unsigned результат по битам один и тот же |
| `sub` | `sub eax, 10` | `eax -= 10` | часто используется и для сравнения через `cmp` |
| `inc` | `inc eax` | `eax++` | короткая форма прибавления 1 |
| `dec` | `dec eax` | `eax--` | короткая форма вычитания 1 |
| `neg` | `neg eax` | `eax = -eax` | двоичное дополнение |
| `imul` | `imul eax, ebx` | signed-умножение | удобная форма: `eax *= ebx` |
| `imul` | `imul eax, ebx, 41` | `eax = ebx * 41` | очень полезно для формул |
| `mul` | `mul ebx` | unsigned-умножение `eax * ebx` | полный результат попадает в `edx:eax` |
| `cdq` | `cdq` | расширяет знак `eax` в `edx:eax` | почти всегда перед `idiv` |
| `idiv` | `idiv dword [y]` | signed-деление `edx:eax / y` | частное в `eax`, остаток в `edx` |
| `div` | `div dword [y]` | unsigned-деление `edx:eax / y` | перед ним часто делают `xor edx, edx` |

Запомни ритуал signed-деления:

```asm
mov eax, [x]
cdq
idiv dword [y]
; eax = x / y
; edx = x % y
```

Для unsigned-деления чаще так:

```asm
mov eax, [x]
xor edx, edx
div dword [y]
```

### 3. Битовые операции

| Команда | Пример | Что делает | Где пригодится |
|---|---|---|---|
| `and` | `and eax, mask` | оставляет биты, где в маске 1 | фильтрация битов, masked merge |
| `or` | `or eax, ebx` | ставит биты из обоих операндов | сборка числа из частей |
| `xor` | `xor eax, ebx` | 1 там, где биты различаются | branchless tricks, обнуление |
| `xor` | `xor eax, eax` | быстро делает `eax = 0` | часто в `return 0` |
| `not` | `not eax` | инвертирует все биты | `~x` |
| `shl` / `sal` | `shl eax, 8` | сдвиг влево | `x << 8`, упаковка байтов |
| `shr` | `shr eax, 1` | логический сдвиг вправо | unsigned `x >> 1` |
| `sar` | `sar eax, 31` | арифметический сдвиг вправо | маска знака для `abs` |
| `rol` | `rol eax, cl` | циклический сдвиг влево | задачи на rotate |
| `ror` | `ror eax, cl` | циклический сдвиг вправо | задача “Поворот” |

Мини-пример из домашних:

```asm
; 01-7: pack = a | (b << 8) | (c << 16) | (d << 24)
mov eax, [a]
mov ecx, [b]
shl ecx, 8
or eax, ecx
```

И ещё один:

```asm
; 01-8: result = (a & c) | (b & ~c)
mov eax, [a]
and eax, [c]
mov ecx, [c]
not ecx
and ecx, [b]
or eax, ecx
```

### 4. Сравнения, флаги и переходы

`cmp` и `test` сами по себе никуда не прыгают. Они только выставляют флаги. Прыгают уже команды `j...`.

| Команда | Пример | Что делает | Как думать |
|---|---|---|---|
| `cmp` | `cmp eax, ebx` | выставляет флаги как после `eax - ebx` | сравнение через вычитание |
| `test` | `test eax, eax` | выставляет флаги как после `eax & eax` | проверка на ноль |
| `jmp` | `jmp .loop` | безусловный переход | `goto` |
| `je` / `jz` | `je .equal` | переход, если равно / ноль | `==` |
| `jne` / `jnz` | `jne .not_equal` | переход, если не равно / не ноль | `!=` |
| `jg` | `jg .greater` | signed `>` | для `int` |
| `jge` | `jge .ge` | signed `>=` | для `int` |
| `jl` | `jl .less` | signed `<` | для `int` |
| `jle` | `jle .le` | signed `<=` | для `int` |
| `ja` | `ja .above` | unsigned `>` | для `unsigned` |
| `jae` | `jae .ae` | unsigned `>=` | для `unsigned` |
| `jb` | `jb .below` | unsigned `<` | для `unsigned` |
| `jbe` | `jbe .be` | unsigned `<=` | для `unsigned` |

Пример:

```asm
cmp eax, ebx
jl .less_signed      ; if ((int)eax < (int)ebx)
jb .less_unsigned    ; if ((unsigned)eax < (unsigned)ebx)
```

Один и тот же набор битов может быть `-1` как signed и `4294967295` как unsigned. Поэтому `jl` и `jb` — не одно и то же.

### 5. Функции и стек

| Команда | Пример | Что делает | Как думать |
|---|---|---|---|
| `call` | `call f` | кладёт адрес возврата на стек и прыгает в `f` | вызов функции |
| `ret` | `ret` | достаёт адрес возврата и прыгает туда | выход из функции |
| `leave` | `leave` | аналог `mov esp, ebp; pop ebp` | быстрый эпилог функции |
| `push` | `push arg` | положить аргумент на стек | подготовка вызова |
| `pop` | `pop eax` | снять значение со стека | восстановление/чтение |

Типовой CDECL-вызов:

```asm
push dword [b]
push dword [a]
call sum
add esp, 8
```

Типовая функция:

```asm
sum:
    push ebp
    mov ebp, esp

    mov eax, [ebp+8]
    add eax, [ebp+12]

    pop ebp
    ret
```

Карта фрейма:

```text
[ebp+12]  second argument
[ebp+8]   first argument
[ebp+4]   return address
[ebp]     old ebp
[ebp-4]   first local variable
```

### 6. Минимум x87, чтобы не испугаться вещественных чисел

x87 — отдельная стековая машина для floating point. Её не надо мешать с `eax/ebx`.

| Команда | Пример | Что делает |
|---|---|---|
| `finit` | `finit` | инициализирует x87 |
| `fld` | `fld dword [x]` | кладёт float на x87-стек |
| `fst` | `fst dword [x]` | сохраняет без pop |
| `fstp` | `fstp qword [esp+4]` | сохраняет и делает pop |
| `faddp` | `faddp` | складывает верхние элементы и делает pop |
| `fsubp` | `fsubp` | вычитает и делает pop |
| `fmulp` | `fmulp` | умножает и делает pop |
| `fdivp` | `fdivp` | делит и делает pop |

Главная ловушка:

```asm
; printf("%f") ждёт double, то есть qword
fstp qword [esp+4]
```

Если передать `dword`, можно получить мусор.

### 7. Очень короткая шпаргалка “что учить первым”

Сначала уверенно выучи и используй вот этот минимум:

