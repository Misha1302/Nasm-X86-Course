# День 19. Структуры: поля — это смещения

## Опора на материалы ВШЭ

`Slides2026-0xB.pdf`: структуры, доступ к полям, массив внутри структуры, linked list, alignment, padding, layout структур в памяти.

## Зачем этот день

В C++ ты пишешь красиво:

```cpp
r->i
r->a[k]
r->p
```

А в памяти нет имён `i`, `a`, `p`. Там просто байты.

Компилятор знает, что поле `i` лежит со смещения 0, поле `a` со смещения 8, поле `p` со смещения 20. В asm это превращается в обычную адресацию:

```asm
[base + offset]
[base + index*scale + offset]
```

Сегодня мы учимся видеть структуру как layout в памяти.

---

## Главная мысль

Структура — это непрерывный блок памяти.

Поле — это фиксированное смещение внутри этого блока.

Padding — это пустые байты, которые компилятор вставляет для выравнивания.

---

## 1. Простая структура

C++:

```cpp
struct Point {
    int x;
    int y;
};
```

В IA-32 `int` занимает 4 байта, значит layout такой:

```text
offset:  0      4
       +------+------+
field: |  x   |  y   |
       +------+------+
```

Если `eax = address of Point`, то:

```asm
mov edx, [eax]      ; edx = point.x
mov edx, [eax + 4]  ; edx = point.y
```

В памяти нет строк “x” и “y”. Есть только смещения.

---

## 2. Структура с массивом и указателем

C++:

```cpp
struct rec {
    int i;
    int j;
    int a[3];
    rec* p;
};
```

В IA-32 pointer занимает 4 байта. Получаем layout:

```text
offset:  0      4      8      12     16     20
       +------+------+------+------+------+------+
field: |  i   |  j   | a[0] | a[1] | a[2] |  p   |
       +------+------+------+------+------+------+
```

Размер структуры:

```text
24 bytes
```

Если `edx = r`, то:

| C++ access | ASM address |
|---|---|
| `r->i` | `[edx]` or `[edx+0]` |
| `r->j` | `[edx+4]` |
| `r->a[0]` | `[edx+8]` |
| `r->a[1]` | `[edx+12]` |
| `r->a[k]` | `[edx+8+4*k]` |
| `r->p` | `[edx+20]` |

---

## 3. Разбор asm из структуры

```asm
mov eax, [edx]                 ; eax = r->i
mov [edx + 4*eax + 8], ecx     ; r->a[i] = val
mov edx, [edx + 20]            ; r = r->p
```

Разбираем.

### `mov eax, [edx]`

`edx` — база структуры.

`[edx]` — поле со смещения 0.

Значит:

```cpp
eax = r->i;
```

### `mov [edx + 4*eax + 8], ecx`

Адрес:

```text
edx + 8 + 4*eax
```

`8` — начало массива `a`.

`4*eax` — индекс элемента `int`.

Значит:

```cpp
r->a[i] = val;
```

если `eax = i`, `ecx = val`.

### `mov edx, [edx + 20]`

Поле со смещения 20 — `p`.

Значит:

```cpp
r = r->p;
```

Это похоже на переход к следующему узлу linked list.

---

## 4. Linked list на уровне asm

C++:

```cpp
while (r != nullptr) {
    sum += r->i;
    r = r->p;
}
```

ASM-shape:

```asm
xor eax, eax          ; sum = 0

.loop:
    test edx, edx     ; r == nullptr?
    je .end

    add eax, [edx]    ; sum += r->i
    mov edx, [edx+20] ; r = r->p
    jmp .loop

.end:
```

Здесь структура стала просто набором смещений:

```text
r->i = [edx]
r->p = [edx+20]
```

## 5. Padding и alignment

Иногда поля не лежат вплотную.

C++:

```cpp
struct S1 {
    char c;
    int x;
    char d;
};
```

Наивно можно подумать:

```text
1 + 4 + 1 = 6 bytes
```

Но компилятор обычно хочет, чтобы `int` лежал по адресу, кратному 4. Поэтому он вставляет padding.

Возможный layout IA-32:

```text
offset: 0   1   2   3   4          8   9  10  11
      +---+---+---+---+------------+---+---+---+---+
      | c | p | p | p |     x      | d | p | p | p |
      +---+---+---+---+------------+---+---+---+---+
```

`p` — padding byte.

Размер может стать 12 байт, а не 6.

---

## 6. Как порядок полей влияет на размер

Если переставить поля:

```cpp
struct S2 {
    int x;
    char c;
    char d;
};
```

Возможный layout:

```text
offset: 0          4   5   6   7
      +------------+---+---+---+---+
      |     x      | c | d | p | p |
      +------------+---+---+---+---+
```

Размер может быть 8 байт.

Поля те же, но структура компактнее.

Практический вывод:

> Порядок полей влияет на размер структуры из-за padding.

---

## 7. Почему нужен padding

Процессорам обычно удобнее читать данные с выровненных адресов.

Грубое правило:

```text
char  -> можно почти где угодно
short -> желательно адрес кратен 2
int   -> желательно адрес кратен 4
```

Это зависит от ABI и архитектуры, но для учебных задач достаточно понимать идею.

Компилятор вставляет padding, чтобы поля лежали по удобным адресам.

## 8. Array of structs

C++:

```cpp
rec arr[10];
```

Если `sizeof(rec) = 24`, то:

```text
&arr[k] = base + 24*k
```

Поле `arr[k].j`:

```text
base + 24*k + 4
```

Поле `arr[k].a[i]`:

```text
base + 24*k + 8 + 4*i
```

В asm масштаб `24*k` нельзя записать напрямую как `index*24`, потому что scale может быть только 1,2,4,8. Компилятор может использовать `imul` или несколько `lea`.

Пример:

```asm
mov eax, [k]
imul eax, 24
mov ecx, [base + eax + 4] ; arr[k].j
```

---

## 9. Union

`union` отличается от `struct` тем, что все поля начинаются с одного offset.

C++:

```cpp
union U {
    int x;
    char bytes[4];
};
```

Layout:

```text
offset 0:
+----------------+
| x / bytes[0..3]|
+----------------+
```

Размер union — размер самого большого поля с учётом выравнивания.

Это полезно для понимания того, что разные типы могут смотреть на одни и те же байты.

---

## 10. Reverse-паттерны структур

| ASM | Возможный C-смысл |
|---|---|
| `[eax]` | первое поле структуры |
| `[eax+4]` | второе `int`-поле или поле со смещением 4 |
| `[eax+20]` | поле со смещением 20, часто pointer в примерах |
| `[eax+4*ecx+8]` | массив `int` внутри структуры с offset 8 |
| `mov eax, [eax+offset]` | переход по pointer-полю |
| `imul index, sizeof(struct)` | доступ к array of structs |

Когда видишь постоянные смещения `0`, `4`, `8`, `20`, думай не только про массивы, но и про структуры.

## 11. Мини-челленджи

#### 1. Объясни адрес

```asm
[edx + 4*eax + 8]
```

если `edx = r`, `eax = i`, а у структуры есть `int a[3]` со смещения 8.

<details>
<summary>Ответ</summary>

Это `r->a[i]`.

</details>

#### 2. Почему `sizeof(S1)` не 6?

```cpp
struct S1 {
    char c;
    int x;
    char d;
};
```

<details>
<summary>Ответ</summary>

Из-за padding/alignment. Компилятор вставляет пустые байты, чтобы `int x` лежал по выровненному адресу и чтобы элементы массива структур тоже были выровнены.

</details>

#### 3. Что хранится в памяти: имена полей или байты?

<details>
<summary>Ответ</summary>

Байты. Имена полей существуют на уровне исходного кода и отладочной информации, но обычный машинный код работает со смещениями.

</details>

#### 4. Адрес `arr[k].j`

Если `sizeof(rec)=24`, `j` имеет offset 4, какой адрес?

<details>
<summary>Ответ</summary>

```text
base + 24*k + 4
```

</details>

#### 5. Что делает `mov edx, [edx+20]` в linked list?

<details>
<summary>Ответ</summary>

Переходит к следующему узлу: `r = r->p`, если поле `p` лежит со смещения 20.

</details>

## 12. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| считать `sizeof(struct)` суммой полей | padding может увеличить размер |
| забыть offset поля | попадёшь не в то поле |
| путать массив структур и структуру с массивом | разные формулы адреса |
| думать, что имена полей есть в памяти | в машинном коде только смещения |
| забыть размер pointer в IA-32 | pointer = 4 bytes |
| записать scale 24 в адресации | x86 scale только 1,2,4,8 |
| не учитывать alignment | неверно посчитаешь layout |
