# День 11. EFLAGS: лампочки после инструкции

## Опора на материалы ВШЭ

`Slides2026-04.pdf`, `Slides2026-06.pdf`: регистр `EFLAGS`, арифметические флаги, влияние инструкций на флаги, подготовка к `cmp/test/jcc`.

## Зачем этот день

В C++ результат условия выглядит как `true` или `false`:

```cpp
if (x == y) { ... }
```

На уровне процессора обычно нет отдельного “булевого результата сравнения”. Вместо этого инструкция оставляет следы в специальном регистре флагов, а условный переход потом смотрит на эти следы.

То есть:

```text
операция -> флаги -> условный переход
```

Сегодня учимся понимать эти флаги. Без этого `cmp`, `test`, `je`, `jl`, `ja` будут выглядеть как магия.

---

## Главная мысль

Флаги — это признаки результата последней операции.

```text
+----+----+----+----+----+----+
| OF | SF | ZF | AF | PF | CF |
+----+----+----+----+----+----+
  |    |    |    |    |    +-- unsigned carry / borrow
  |    |    |    |    +------- parity of low byte
  |    |    |    +------------ auxiliary carry
  |    |    +----------------- zero result
  |    +---------------------- sign bit of result
  +--------------------------- signed overflow
```

Флаги живут до следующей инструкции, которая их изменит.

Это очень важно: условный переход смотрит не “на старую операцию вообще”, а на текущее состояние флагов.

---

## `ZF`: zero flag

`ZF = 1`, если результат операции равен нулю.

Пример:

```asm
mov eax, 5
sub eax, 5
```

После `sub`:

```text
eax = 0
ZF = 1
```

Ещё пример:

```asm
xor eax, eax
```

После него:

```text
eax = 0
ZF = 1
```

Поэтому `test eax, eax` + `je` часто используется для проверки на ноль. Подробно будет в следующем дне.

---

## `SF`: sign flag

`SF` равен старшему биту результата.

Для 32-битного результата:

```text
SF = bit 31
```

Если старший бит 1, signed-интерпретация обычно отрицательная.

Пример:

```text
result = 0xFFFFFFFF
SF = 1
```

Но осторожно: `SF=1` не всегда достаточно, чтобы сделать вывод “меньше” после сравнения. Для signed-сравнений ещё важен `OF`.

---

## `CF`: carry flag

`CF` показывает перенос или заём для unsigned-арифметики.

8-битный пример:

```text
  11111111   255
+ 00000001     1
-----------
1 00000000
```

В 8 битах сохранился ноль, а перенос ушёл наружу.

```text
CF = 1
```

Для unsigned это значит: результат не поместился.

Ещё пример с вычитанием:

```text
0 - 1
```

Для unsigned нужно занять за пределами диапазона, значит тоже будет borrow, то есть `CF=1`.

---

## `OF`: overflow flag

`OF` показывает signed overflow.

Signed 8-bit диапазон:

```text
-128..127
```

Пример:

```text
  01111111   127
+ 00000001     1
-----------
  10000000   -128 as signed
```

Математически должно быть `128`, но signed 8-bit не может это представить.

```text
OF = 1
```

Главное различие:

```text
CF -> unsigned overflow / carry
OF -> signed overflow
```

---

## `PF` и `AF`

Эти флаги реже нужны в обычных задачах, но в слайдах они есть, поэтому знать их стоит.

| Flag | Смысл |
|---|---|
| `PF` | parity flag: чётность количества единичных битов в младшем байте результата |
| `AF` | auxiliary carry: перенос между младшей и старшей половиной младшего байта; исторически важен для BCD |

Для экзамена чаще всего реально нужны:

```text
ZF, SF, CF, OF
```

Но если спрашивают “какие флаги есть в EFLAGS”, `PF` и `AF` тоже надо назвать.

---

## Какие инструкции меняют флаги

Упрощённая таблица:

| Инструкция | Что с флагами |
|---|---|
| `add`, `sub`, `cmp`, `neg` | обновляют арифметические флаги |
| `and`, `or`, `xor`, `test` | обновляют `ZF/SF/PF`, обычно сбрасывают `CF/OF` |
| `inc`, `dec` | обновляют почти как `add/sub`, но не меняют `CF` |
| `mov`, `lea` | не меняют флаги |
| `push`, `pop` | обычно не меняют флаги |
| `jmp`, `call`, `ret` | сами по себе не являются арифметикой; флаги не используют как результат вычисления |

Важное практическое правило:

> Если после `cmp` ты хочешь сделать `je`, не вставляй между ними инструкцию, которая меняет флаги.

---

## Главная ловушка: флаги легко перезаписать

Плохо:

```asm
cmp eax, ebx
add ecx, 1
je .equal
```

Почему плохо?

`je` должен был смотреть на результат `cmp eax, ebx`, но между ними стоит `add ecx, 1`. `add` изменил флаги. Теперь `je` смотрит уже на результат `add`, а не на сравнение.

Хорошо:

```asm
cmp eax, ebx
je .equal
add ecx, 1
```

Или так, если надо сохранить порядок действий:

```asm
add ecx, 1
cmp eax, ebx
je .equal
```

---

## `mov` не меняет флаги

Это полезно.

```asm
cmp eax, ebx
mov edx, 123
je .equal
```

`mov` не меняет флаги, поэтому `je` всё ещё смотрит на результат `cmp`.

Но такой стиль всё равно лучше использовать аккуратно: чем ближе `jcc` к `cmp/test`, тем проще читать код.

---

## Мини-примеры

### `xor eax, eax`

```asm
xor eax, eax
```

Результат:

```text
eax = 0
ZF = 1
CF = 0
OF = 0
```

Почему `CF/OF = 0`? Потому что логические операции вроде `xor` сбрасывают carry/overflow.

### `sub eax, eax`

```asm
sub eax, eax
```

Тоже даёт:

```text
eax = 0
ZF = 1
```

Но `xor eax, eax` обычно короче и идиоматичнее для обнуления.

### `test eax, eax`

```asm
test eax, eax
```

Значение `eax` не меняется, но флаги обновляются как после `eax & eax`.

Если `eax = 0`, будет `ZF=1`.

## Мини-челленджи

### 1. Что делает с флагами `xor eax, eax`?

<details>
<summary>Ответ</summary>

Делает `eax = 0`, ставит `ZF = 1`, сбрасывает `CF` и `OF`.

</details>

### 2. Почему опасно вставлять `add` между `cmp` и `je`?

<details>
<summary>Ответ</summary>

Потому что `add` изменит флаги, и `je` будет смотреть уже не на результат `cmp`.

</details>

### 3. Чем `CF` отличается от `OF`?

<details>
<summary>Ответ</summary>

`CF` нужен для unsigned-переноса/заёма. `OF` нужен для signed overflow.

</details>

### 4. Меняет ли `mov eax, ebx` флаги?

<details>
<summary>Ответ</summary>

Нет.

</details>

### 5. Что означает `ZF=1`?

<details>
<summary>Ответ</summary>

Результат последней флагоизменяющей операции был равен нулю.

</details>

## Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что флаги вечные | следующая инструкция может их изменить |
| вставлять `add/sub/and/xor` между `cmp` и `jcc` | переход будет смотреть не на те флаги |
| путать `CF` и `OF` | один unsigned, другой signed |
| думать, что `SF` один решает signed comparison | нужен ещё `OF` |
| считать, что `mov` обновляет флаги | `mov` флаги не меняет |
