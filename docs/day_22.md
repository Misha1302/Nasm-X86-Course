# День 22. Вещественные числа: почему `0.1` не точное

## Опора на материалы ВШЭ

`Slides2026-0xF.pdf`, `Slides2026-15.pdf`: двоичные дроби, floating point, нормализованные и денормализованные числа, `+0/-0`, `Inf`, `NaN`, округление и ограничения представления.

## Зачем этот день

С целыми числами всё было относительно честно:

```text
32 бита -> одно из 2^32 возможных значений
```

Да, signed/unsigned могут читать эти биты по-разному, но сами целые значения в диапазоне представлены точно.

С вещественными числами иначе. Многие привычные десятичные дроби нельзя точно записать конечным числом двоичных битов.

Например:

```cpp
double x = 0.1;
```

обычно хранит не ровно `0.1`, а ближайшее представимое значение.

Сегодня разбираем, почему так происходит и как floating point устроен на уровне идеи.

---

## Главная мысль

Floating point — это приближённое представление числа в форме:

```text
sign * mantissa * 2^exponent
```

У числа есть:

- знак;
- порядок / exponent;
- дробная часть / fraction;
- специальные случаи: нули, денормалы, бесконечности, NaN.

---

## 1. Двоичные дроби

В десятичной системе:

```text
123.45 = 1*100 + 2*10 + 3*1 + 4/10 + 5/100
```

В двоичной системе дробная часть использует степени 2:

```text
0.1₂ = 1/2
0.01₂ = 1/4
0.001₂ = 1/8
```

Пример:

```text
5.75 decimal
```

Целая часть:

```text
5 = 101₂
```

Дробная часть:

```text
0.75 = 1/2 + 1/4 = 0.11₂
```

Итого:

```text
5.75₁₀ = 101.11₂
```

---

## 2. Почему `0.1` не точное

`0.1` в десятичной системе — это:

```text
1/10
```

А:

```text
10 = 2 * 5
```

Конечные двоичные дроби имеют знаменатель степени двойки:

```text
1/2, 1/4, 1/8, 1/16, ...
```

Если в знаменателе есть множитель 5, дробь в binary обычно становится бесконечной периодической.

```text
0.1₁₀ = 0.00011001100110011...₂
```

Поэтому компьютер хранит ближайшее число, которое помещается в доступные биты.

---

## 3. Почему это важно

В C++:

```cpp
double a = 0.1;
double b = 0.2;
double c = 0.3;
```

Не стоит уверенно писать:

```cpp
if (a + b == c)
```

Потому что `a`, `b`, `c` могут быть приближениями.

Обычно сравнивают с погрешностью:

```cpp
abs((a + b) - c) < eps
```

Ассемблер здесь не делает магию точности. Он просто выполняет операции над представимыми floating point значениями.

---

## 4. Общая идея scientific notation

В десятичной записи:

```text
12345 = 1.2345 * 10^4
```

В двоичной floating point идея похожая:

```text
value = mantissa * 2^exponent
```

Например:

```text
101.11₂ = 1.0111₂ * 2^2
```

Потому что мы перенесли точку на 2 позиции влево.

---

## 5. Layout `float`

Обычный 32-битный `float` можно представить так:

```text
31     30          23 22                    0
+--------+------------+----------------------+
| sign   | exponent   | fraction             |
+--------+------------+----------------------+
  1 bit     8 bits           23 bits
```

| Поле | Зачем |
|---|---|
| `sign` | знак числа |
| `exponent` | порядок, степень двойки с bias |
| `fraction` | дробная часть мантиссы |

Упрощённая формула:

```text
(-1)^sign * mantissa * 2^exponent
```

Точная IEEE-формула использует bias и скрытую единицу для нормализованных чисел, но на первом проходе важна идея: число хранится не как десятичная строка, а как знак + масштаб + значащие биты.

---

## 6. Нормализованные числа

Для обычных нормализованных чисел мантисса имеет вид:

```text
1.xxxxx₂
```

Эта первая `1` обычно не хранится явно. Её называют hidden bit / implicit leading one.

Почему можно не хранить?

Потому что у нормализованного ненулевого двоичного числа первая значащая цифра всегда `1`.

Пример:

```text
101.11₂ = 1.0111₂ * 2^2
```

Мантисса начинается с `1.`.

---

## 7. Денормализованные числа

Денормализованные числа нужны для очень маленьких значений около нуля.

Идея:

```text
когда нормализованного exponent уже не хватает,
мы разрешаем числа без скрытой ведущей 1.
```

Они помогают сделать переход к нулю более плавным.

Для курса достаточно понимать:

- normal — обычные числа;
- denormal/subnormal — очень маленькие числа около нуля;
- они имеют отдельный encoding.

---

## 8. Специальные значения

Floating point хранит не только обычные числа.

| exponent | fraction | Meaning |
|---|---|---|
| all zero | zero | `+0` / `-0` |
| all zero | non-zero | denormalized/subnormal |
| normal | any | normalized number |
| all ones | zero | `+Inf` / `-Inf` |
| all ones | non-zero | `NaN` |

### `+0` и `-0`

У floating point есть положительный и отрицательный ноль.

Обычно они сравниваются как равные, но в некоторых операциях знак нуля может проявляться.

### Infinity

`Inf` появляется, например, при overflow или делении ненулевого числа на ноль в floating point модели.

### NaN

`NaN` = Not a Number.

Это не “очень большое число”. Это специальное значение для некорректных результатов:

```text
0.0 / 0.0
sqrt(-1.0) в вещественной арифметике
Inf - Inf
```

---

## 9. Округление

Так как битов конечное число, результат операции часто приходится округлять.

Например, если точный результат требует больше битов мантиссы, чем есть, процессор выбирает ближайшее представимое значение по текущему режиму округления.

Типовая мысль:

```text
floating point arithmetic is not real-number arithmetic
```

Она похожа на вещественную математику, но не равна ей идеально.

---

## 10. Почему порядок операций важен

В вещественной математике:

```text
(a + b) + c = a + (b + c)
```

В floating point это может быть не совсем так из-за округления.

Примерная идея:

```cpp
double a = 1e20;
double b = -1e20;
double c = 3.14;
```

```cpp
(a + b) + c = 3.14
```

а:

```cpp
a + (b + c)
```

может потерять `c` при сложении очень большого и маленького числа.

Для asm это важно потому, что низкоуровневый порядок операций может влиять на точность.

## 12. Мини-челленджи

#### 1. Представь `5.75` в binary

<details>
<summary>Ответ</summary>

```text
5 = 101₂
0.75 = 0.11₂
5.75 = 101.11₂
```

</details>

#### 2. Почему `1/10` бесконечна в binary?

<details>
<summary>Ответ</summary>

Потому что `10 = 2*5`, а конечные двоичные дроби имеют знаменатель степени 2. Множитель 5 даёт бесконечную периодическую запись.

</details>

#### 3. Что означает exponent all ones и fraction zero?

<details>
<summary>Ответ</summary>

Infinity: `+Inf` или `-Inf` в зависимости от sign.

</details>

#### 4. Почему NaN — не “очень большое число”?

<details>
<summary>Ответ</summary>

Потому что NaN означает Not a Number, специальный результат некорректной операции.

</details>

#### 5. Почему `a+b==c` для double может быть плохой проверкой?

<details>
<summary>Ответ</summary>

Потому что `a`, `b`, `c` могут быть приближёнными значениями, и из-за округления точное равенство может не выполниться.

</details>

## 13. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что `0.1` хранится точно | в binary это бесконечная дробь |
| сравнивать float/double через `==` без мысли о погрешности | округление может дать неожиданный результат |
| считать NaN большим числом | NaN — не число |
| забывать про `Inf` | overflow может дать специальное значение |
| думать, что порядок операций всегда не важен | floating point не полностью ассоциативен |
| путать normal и denormal | denormal — специальная зона очень малых чисел |
