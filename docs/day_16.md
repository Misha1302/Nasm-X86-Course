## Главная мысль

Стек IA-32 растёт к меньшим адресам.

`esp` указывает на верхушку стека.

```text
higher addresses
+------------------+
| older stack data |
+------------------+
|                  |
+------------------+ <- esp
lower addresses
```

Когда мы кладём значение на стек, `esp` уменьшается.

Когда снимаем значение со стека, `esp` увеличивается.

---

# 1. Что такое стек

Стек — это структура LIFO:

```text
last in, first out
```

То есть последним положили — первым достали.

Пример:

```asm
push 10
push 20
pop eax
pop ebx
```

Результат:

```text
eax = 20
ebx = 10
```

Потому что `20` лежало на верхушке.

---

# 2. Почему стек растёт вниз

В IA-32 стек растёт в сторону меньших адресов.

Это может быть непривычно: “кладём значение”, а адрес уменьшается.

Но для x86 это норма.

```text
Before push eax:

0x100C | old data |
0x1008 | old data | <- esp

After push eax:

0x100C | old data |
0x1008 | old data |
0x1004 |   eax    | <- esp
```

То есть:

```text
push -> esp -= 4
```

---

# 3. `push`

```asm
push eax
```

примерно значит:

```text
esp = esp - 4
[esp] = eax
```

Почему 4?

Потому что в IA-32 обычный `push` кладёт 32-битное значение, то есть `dword`.

Пример:

```text
eax = 123
esp = 0x1008
```

После:

```asm
push eax
```

получим:

```text
esp = 0x1004
[0x1004] = 123
```

---

# 4. `pop`

```asm
pop eax
```

примерно значит:

```text
eax = [esp]
esp = esp + 4
```

Пример:

```text
esp = 0x1004
[0x1004] = 123
```

После:

```asm
pop eax
```

получим:

```text
eax = 123
esp = 0x1008
```

Важно: `pop` не обязательно “стирает” память. Он просто двигает `esp`. Старые байты могут физически оставаться в памяти, но они больше не считаются верхушкой стека.

---

# 5. Баланс стека

Если ты сделал:

```asm
push eax
push ebx
```

то стек уменьшился на 8 байт.

Чтобы вернуться к прежнему `esp`, нужно либо:

```asm
pop ebx
pop eax
```

либо, если значения не нужны:

```asm
add esp, 8
```

Именно поэтому после:

```asm
push x
push fmt
call scanf
```

мы делаем:

```asm
add esp, 8
```

Аргументы больше не нужны, их можно убрать со стека.

---

# 6. `call`: прыжок с запиской

`call f` — это не просто `jmp f`.

`jmp` просто прыгает.

`call` прыгает и сохраняет адрес возврата.

```text
call f:
1. push address_after_call
2. jump to f
```

То есть:

```asm
call f
```

мысленно похоже на:

```asm
push return_address
jmp f
```

Только процессор делает это сам.

---

# 7. `ret`: вернуться по записке

```asm
ret
```

примерно значит:

```text
1. pop return_address
2. jump to return_address
```

То есть `ret` берёт адрес с вершины стека и передаёт управление туда.

Поэтому если стек испорчен, `ret` может прыгнуть куда угодно.

Это одна из причин, почему стек настолько важен для безопасности.

---

# 8. Как выглядит стек при `call`

Перед:

```asm
call f
```

```text
esp -> [ old stack data ]
```

Сразу внутри `f`:

```text
esp -> [ return address ]
       [ old stack data  ]
```

Потом функция может сама класть на стек локальные данные, сохранять регистры и т.д.

---

# 9. Функция без аргументов

Простейшая функция:

```asm
f:
    mov eax, 123
    ret
```

Вызов:

```asm
call f
; после возврата eax = 123
```

Здесь `eax` используется как возвращаемое значение — это обычная CDECL-идея, которую подробно разберём в следующем дне.

---

# 10. Почему рекурсия требует стек

C++:

```cpp
int fact(int n) {
    if (n <= 1) return 1;
    return n * fact(n - 1);
}
```

Каждый вызов `fact` должен помнить:

- своё значение `n`;
- куда вернуться;
- временные значения;
- иногда сохранённые регистры.

Если вызовы вложенные, всё это складывается на стек:

```text
fact(5)
  fact(4)
    fact(3)
      fact(2)
        fact(1)
```

Каждый уровень имеет свой кусок стека.

---

# 11. Мини-трассировка

Код:

```asm
push 1
push 2
push 3
pop eax
pop ebx
```

После трёх `push`:

```text
esp -> [ 3 ]
       [ 2 ]
       [ 1 ]
```

После `pop eax`:

```text
eax = 3
esp -> [ 2 ]
       [ 1 ]
```

После `pop ebx`:

```text
ebx = 2
esp -> [ 1 ]
```

`1` всё ещё лежит на стеке.

---

# 12. Что нельзя делать со стеком

## Плохо: лишний `pop`

```asm
push eax
pop ebx
pop ecx ; лишний pop
```

Второй `pop` снимет что-то, что ты не клал специально. Это может быть старое значение, адрес возврата или часть чужого фрейма.

## Плохо: забыть очистить аргументы

```asm
push x
push fmt
call printf
; забыли add esp, 8
```

`esp` остаётся ниже, чем должен. Следующие вызовы/возвраты могут работать неправильно.

## Плохо: менять `esp` без понимания

```asm
sub esp, 100
; ...
ret
```

Если не вернуть `esp` обратно, `ret` возьмёт адрес возврата не оттуда.

# 13. Мини-челленджи

### 1. Нарисуй стек

После:

```asm
push 1
push 2
push 3
```

что лежит на верхушке?

<details>
<summary>Ответ</summary>

На верхушке лежит `3`, ниже `2`, ниже `1`.

</details>

### 2. Что будет в регистрах?

```asm
push 10
push 20
pop eax
pop ebx
```

<details>
<summary>Ответ</summary>

```text
eax = 20
ebx = 10
```

</details>

### 3. Объясни `call` двумя действиями

<details>
<summary>Ответ</summary>

`call` кладёт адрес возврата на стек и переходит к функции.

</details>

### 4. Объясни `ret` двумя действиями

<details>
<summary>Ответ</summary>

`ret` снимает адрес возврата со стека и прыгает по нему.

</details>

### 5. Почему `add esp, 8` после двух аргументов?

<details>
<summary>Ответ</summary>

Каждый аргумент занимает 4 байта. Два аргумента — 8 байт.

</details>

# 14. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что стек растёт вверх | в IA-32 `push` уменьшает `esp` |
| забывать, что `push/pop` меняют `esp` | ломается баланс стека |
| делать лишний `pop` | можно снять адрес возврата или чужие данные |
| забывать `add esp, ...` после CDECL-вызова | стек остаётся загрязнённым аргументами |
| менять `esp` и не восстанавливать | `ret` возьмёт адрес не оттуда |
| думать, что `call` — это просто `jmp` | `call` ещё сохраняет адрес возврата |
