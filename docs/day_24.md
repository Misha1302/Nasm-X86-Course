# День 24. C++-объект как `struct` с секретным указателем

## Опора на материалы ВШЭ

`Slides2026-0xD.pdf`: объект как структура, служебные поля, `vptr`, VMT, RTTI, virtual methods, hidden `this`, `thiscall` и низкоуровневая модель C++-объектов.

## Зачем этот день

C++ выглядит высокоуровнево:

```cpp
obj.field
obj.method()
obj.virtualMethod()
```

Но процессор не знает “методов”, “классов” и “виртуальности”. В памяти объект всё равно похож на блок байтов. Методы — это функции. Поля — смещения. Виртуальные методы — косвенные вызовы через таблицу.

Сегодня мы не пишем сложный C++ ABI руками. Мы учимся видеть идею:

```text
object = struct-like layout + sometimes service pointers
method = function with hidden this
virtual call = call through function pointer table
```

---

## Главная мысль

C++-объект на низком уровне похож на структуру.

Если есть виртуальные методы, в объекте обычно появляется скрытый указатель `vptr`, который ведёт к таблице виртуальных методов.

```text
Object:

+------------------+
| vptr             | ---> +---------------------+
+------------------+      | virtual method #0   |
| field1           |      +---------------------+
+------------------+      | virtual method #1   |
| field2           |      +---------------------+
+------------------+
```

---

## 1. Обычный объект без virtual

C++:

```cpp
struct Point {
    int x;
    int y;

    int sum() {
        return x + y;
    }
};
```

Данные объекта похожи на структуру:

```text
offset:  0      4
       +------+------+
field: |  x   |  y   |
       +------+------+
```

Метод `sum` не лежит внутри каждого объекта как отдельный код. Код метода существует отдельно, как функция.

Объект хранит поля, а не копии методов.

---

## 2. Hidden `this`

Когда ты пишешь:

```cpp
p.sum()
```

на низком уровне это похоже на вызов функции с дополнительным скрытым аргументом:

```cpp
Point_sum(&p)
```

`this` — это адрес текущего объекта.

Метод:

```cpp
int Point::sum() {
    return x + y;
}
```

можно мысленно представить так:

```cpp
int Point_sum(Point* this) {
    return this->x + this->y;
}
```

В asm это превращается в доступы по смещениям от адреса объекта:

```asm
mov eax, [this + 0]  ; x
add eax, [this + 4]  ; y
```

Конкретно где передаётся `this`, зависит от соглашения вызова и компилятора. В 32-битных C++ ABI часто встречается `thiscall`, где `this` может передаваться через регистр, например `ecx`, но детали ABI лучше уточнять под платформу.

---

## 3. Поля — это offsets

C++:

```cpp
class A {
public:
    int x;
    int y;
};
```

Если `eax = &obj`, то:

```asm
mov edx, [eax]      ; obj.x
mov edx, [eax + 4]  ; obj.y
```

Никакого имени `x` в обычной памяти объекта нет.

Имя `x` использовалось компилятором, чтобы выбрать offset `0`.

Имя `y` использовалось компилятором, чтобы выбрать offset `4`.

## 4. Virtual methods и `vptr`

Если в классе есть виртуальные методы:

```cpp
struct Base {
    virtual int f();
    int x;
};
```

объекту обычно нужен способ понять, какую именно реализацию `f` вызвать.

Для этого используется таблица виртуальных методов.

Упрощённо:

```text
Base object:

+------------------+
| vptr             | ---> VMT / vtable
+------------------+
| x                |
+------------------+
```

`vptr` — указатель на таблицу.

Таблица содержит адреса виртуальных функций.

```text
vtable:
+---------------------+
| &Base::f            |
+---------------------+
| &Base::otherVirtual |
+---------------------+
```

## 5. Virtual call

C++:

```cpp
Base* p = ...;
p->f();
```

Если `f` виртуальная, компилятор не может просто написать:

```asm
call Base_f
```

Потому что реальный объект может быть наследником:

```cpp
struct Derived : Base {
    int f() override;
};
```

Тогда нужно вызвать `Derived::f`, если объект на самом деле `Derived`.

Упрощённый asm-shape:

```asm
mov eax, [p]        ; eax = object pointer, условно
mov edx, [eax]      ; edx = vptr
call dword [edx]    ; call first virtual method
```

Для метода не на первой позиции таблицы будет смещение:

```asm
call dword [edx + 4]
```

В реальном коде будет ещё передача `this` и ABI-детали.

---

## 6. Direct call vs indirect call

Обычный вызов функции:

```asm
call some_function
```

Адрес функции известен прямо в коде.

Виртуальный вызов:

```asm
call dword [edx + offset]
```

Адрес функции читается из памяти, из таблицы.

| Вызов | Как выбирается адрес |
|---|---|
| direct call | адрес функции известен заранее |
| indirect call | адрес функции лежит в регистре/памяти |
| virtual call | адрес берётся из vtable объекта |

## 7. Inheritance на уровне идеи

C++:

```cpp
struct Base {
    int a;
};

struct Derived : Base {
    int b;
};
```

Упрощённо объект `Derived` может начинаться с части `Base`:

```text
Derived object:

+------------------+
| Base::a          |
+------------------+
| Derived::b       |
+------------------+
```

Поэтому указатель на `Derived` часто можно рассматривать как указатель на `Base`-часть объекта.

Для сложного наследования, multiple inheritance и virtual inheritance layout становится сложнее. Для этого курса достаточно базовой идеи: объект — это layout, а наследование влияет на расположение полей и служебных указателей.

## 8. RTTI

RTTI = runtime type information.

Это служебная информация, которая помогает во время выполнения узнавать тип объекта, например для:

```cpp
dynamic_cast

typeid
```

RTTI часто связана с таблицами виртуальных методов и служебными структурами компилятора.

Для курса достаточно понимать:

```text
RTTI — не магия языка, а дополнительные данные runtime/ABI.
```

---

## 9. Почему это важно для asm

Если ты увидел:

```asm
mov eax, [ecx]
call dword [eax+8]
```

это может быть виртуальный вызов:

```text
ecx = this
eax = vptr
[eax+8] = address of virtual method
```

Если ты увидел доступы:

```asm
mov eax, [ecx+4]
add eax, [ecx+8]
```

это может быть метод, который читает поля объекта через `this`.

## 10. Что где хранится

| C++ feature | Low-level idea |
|---|---|
| field | bytes at fixed offset |
| object | struct-like memory block |
| non-virtual method | ordinary function with hidden `this` |
| virtual method | function pointer selected from vtable |
| `this` | pointer to current object |
| inheritance | object layout includes base part |
| RTTI | runtime metadata |
| method name | not stored in each object |

## 11. Мини-челленджи

#### 1. Нарисуй объект

Класс:

```cpp
struct A {
    virtual void f();
    int x;
    int y;
};
```

Нарисуй возможный layout.

<details>
<summary>Ответ</summary>

Упрощённо:

```text
+------------------+
| vptr             |
+------------------+
| x                |
+------------------+
| y                |
+------------------+
```

</details>

#### 2. Что такое hidden `this`?

<details>
<summary>Ответ</summary>

Скрытый параметр метода: адрес объекта, для которого вызван метод.

</details>

#### 3. Чем virtual call отличается от прямого `call label`?

<details>
<summary>Ответ</summary>

При virtual call адрес функции берётся из таблицы виртуальных методов объекта, а при direct call адрес известен прямо в инструкции.

</details>

#### 4. Хранятся ли методы внутри каждого объекта?

<details>
<summary>Ответ</summary>

Нет. В объекте хранятся поля и, возможно, служебные указатели. Код методов находится отдельно.

</details>

#### 5. Что может означать `mov eax, [ecx+4]` внутри метода?

<details>
<summary>Ответ</summary>

Чтение поля объекта по offset 4, если `ecx` содержит `this`.

</details>

## 12. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что объект хранит копии методов | методы — код отдельно, объект хранит данные |
| считать `vptr` обычным видимым полем C++ | это служебная деталь ABI/реализации |
| забывать hidden `this` | методы не поймут, с каким объектом работать |
| думать, что virtual call — обычный `call label` | он идёт через таблицу/указатель |
| переносить layout между компиляторами без проверки | C++ ABI и layout могут отличаться |
| слишком буквально воспринимать упрощённые схемы | реальный C++ object model сложнее |
