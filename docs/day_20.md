# День 20. Что происходит до `main` и после него

## Опора на материалы ВШЭ

`Slides2026-0xC.pdf`: `_start`, `__libc_start_main`, initial stack, `argc/argv/envp`, `.init_array`, `.fini_array`, `objdump`, этапы сборки и запуска программы.

## Зачем этот день

В C++ кажется, что программа начинается с `main`.

```cpp
int main() {
    return 0;
}
```

Это удобная иллюзия. На самом деле до `main` уже произошло довольно много событий:

- ОС загрузила executable;
- подготовила память процесса;
- подготовила начальный стек;
- стартовый код вызвал libc;
- libc вызвала init-функции;
- и только потом управление дошло до твоего `main`.

Сегодня мы убираем последнюю большую магию: **что значит “программа запустилась”**.

---

## Главная мысль

`main` — это не первая инструкция процесса.

`main` — это пользовательская функция, которую вызывает стартовый код runtime.

Упрощённый путь:

```text
OS loader
   |
   v
_start
   |
   v
__libc_start_main(...)
   |
   v
constructors / init
   |
   v
main(argc, argv, envp)
   |
   v
destructors / fini
   |
   v
exit
```

Когда мы пишем:

```asm
global main
```

мы говорим не “вот первая инструкция процесса”, а:

> вот функция `main`, которую C runtime должен найти и вызвать.

---

# 1. Ещё раз: путь сборки

До запуска программа проходит несколько стадий.

```text
source code
   |
   | preprocessing / compilation
   v
assembly
   |
   | assembling
   v
object file (.o)
   |
   | linking
   v
executable
   |
   | loading
   v
process in memory
```

Для NASM:

```bash
nasm -f elf32 main.asm -o main.o
gcc -m32 main.o -o main
./main
```

Стадии:

| Stage | Что получается |
|---|---|
| assembling | object file `.o` |
| linking | executable |
| loading | процесс в памяти |
| startup | `_start`, libc, init |
| user code | `main` |

---

# 2. Что делает loader

Loader — часть ОС, которая загружает программу в память.

Он примерно делает:

- читает executable file;
- создаёт адресное пространство процесса;
- размещает секции кода и данных;
- подключает динамические библиотеки;
- готовит стек;
- передаёт управление стартовой точке программы.

Упрощённая память процесса:

```text
higher addresses
+--------------------------+
| stack                    |
| grows downward           |
+--------------------------+
|                          |
| mmap / shared libraries  |
|                          |
+--------------------------+
| heap                     |
| grows upward             |
+--------------------------+
| .bss                     |
+--------------------------+
| .data                    |
+--------------------------+
| .rodata                  |
+--------------------------+
| .text                    |
+--------------------------+
lower addresses
```

Это не точная карта для каждой системы, но хорошая учебная модель.

---

# 3. Начальный стек

Когда программа только стартует, ОС кладёт на стек данные запуска.

Обычно там можно найти:

- `argc` — количество аргументов командной строки;
- `argv` — массив строк аргументов;
- `envp` — переменные окружения;
- служебные данные для dynamic loader.

Пример запуска:

```bash
./main hello 123
```

Тогда C видит:

```c
argc = 3
argv[0] = "./main"
argv[1] = "hello"
argv[2] = "123"
```

В `main` это приходит уже в удобном виде:

```c
int main(int argc, char** argv, char** envp)
```

Но до этого стартовый код должен забрать данные из начального стека и вызвать `main` по правилам ABI.

---

# 4. `_start`

`_start` — реальная стартовая точка обычной C-программы.

Ты обычно её не пишешь. Она приходит из startup files, которые подключает linker через `gcc`.

Именно поэтому мы линковали так:

```bash
gcc -m32 main.o -o main
```

а не пытались вручную сразу сделать всё через `ld`.

`_start` подготавливает вызов libc и в итоге приводит к вызову `main`.

Если собрать совсем без libc и самому написать `_start`, можно сделать программу без `main`, но это уже отдельная низкоуровневая история.

---

# 5. `__libc_start_main`

`__libc_start_main` — функция libc, которая помогает запустить C-программу правильно.

Она отвечает за такие вещи:

- подготовить окружение C runtime;
- вызвать init-функции;
- вызвать твой `main`;
- обработать результат `main`;
- вызвать финализацию и `exit`.

То есть твой `main` находится внутри более большой схемы.

```text
_start -> __libc_start_main -> main -> exit
```

Для учебной цели не нужно знать все внутренности libc. Важно понимать, почему в дизассемблировании есть код до `main`.

---

# 6. `.init_array` и `.fini_array`

В C++ до `main` могут выполняться конструкторы глобальных объектов.

После `main` могут выполняться деструкторы.

Для этого используются специальные секции, например:

```text
.init_array
.fini_array
```

Упрощённо:

```text
.init_array  -> функции, которые надо вызвать до main
.fini_array  -> функции, которые надо вызвать после main / при exit
```

Пример C++:

```cpp
struct A {
    A()  { /* before main */ }
    ~A() { /* after main */ }
};

A global;

int main() {
    return 0;
}
```

Конструктор `global` выполнится до входа в `main`.

---

# 7. Что показывает `objdump`

Команды:

```bash
objdump -h main
```

Показывает секции.

```bash
objdump -d -M intel main
```

Показывает дизассемблирование.

```bash
objdump -s -j .data main
```

Показывает содержимое `.data`.

```bash
objdump -s -j .init_array main
objdump -s -j .fini_array main
```

Показывает содержимое init/fini секций, если они есть.

---

# 8. Почему в `objdump` много “лишнего”

Когда ты дизассемблируешь программу:

```bash
objdump -d -M intel main
```

ты видишь не только свой `main`.

Там могут быть:

- `_start`;
- PLT/GOT-код для динамических вызовов;
- startup/teardown код;
- служебные функции libc;
- выравнивание;
- куски, сгенерированные компилятором/линкером.

Это нормально.

Для начала ищи свою функцию:

```text
<main>:
```

В `less` удобно:

```text
/main
```

---

# 9. Почему `global main` работает

В NASM:

```asm
global main

main:
    xor eax, eax
    ret
```

`global main` экспортирует символ `main`, чтобы linker и C runtime могли его найти.

Если забыть `global main`, линковщик может сказать что-то вроде:

```text
undefined reference to main
```

Потому что startup-код ожидает, что пользовательская функция `main` существует.

---

# 10. Что если написать `_start` самому

Можно сделать программу без libc:

```asm
section .text
global _start

_start:
    mov eax, 1      ; sys_exit on old 32-bit Linux syscall ABI
    xor ebx, ebx    ; status 0
    int 0x80
```

Но в этом курсе основной путь другой:

```text
global main + gcc -m32 + libc
```

Почему?

Потому что задачи используют `scanf`, `printf`, CDECL и обычное C-окружение. Нам выгоднее понимать `main`, а не вручную писать весь runtime.

# 11. Мини-челленджи

### 1. Найди `_start`

Собери программу и выполни:

```bash
objdump -d -M intel main | less
```

Найди `<_start>`.

<details>
<summary>Подсказка</summary>

В `less` нажми `/`, введи `_start`, Enter.

</details>

### 2. Найди `main`

Найди `<main>` в том же дизассемблировании.

<details>
<summary>Ответ</summary>

Там будет твой код: `xor eax,eax`, `ret` или инструкции из твоей программы.

</details>

### 3. Почему `main` не первая инструкция?

<details>
<summary>Ответ</summary>

Потому что сначала управление получает стартовый код (`_start`), который подготавливает C runtime и вызывает `main`.

</details>

### 4. Что такое `.init_array`?

<details>
<summary>Ответ</summary>

Секция с адресами функций, которые runtime вызывает до `main`, например для глобальных конструкторов C++.

</details>

### 5. Почему мы используем `gcc -m32` для линковки?

<details>
<summary>Ответ</summary>

Потому что `gcc` как linker driver подключает нужный startup-код и libc, а `-m32` говорит линковать 32-битную программу.

</details>

# 12. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что `main` — первая инструкция процесса | сначала работает loader/startup/runtime |
| искать только свой код в `objdump` и пугаться лишнего | дизассемблируется весь executable |
| забыть `global main` | startup-код не найдёт `main` |
| путать `.o` и executable | object file ещё не запустишь как программу |
| линковать через `ld` без понимания startup/libc | легко получить ошибки или неправильный запуск |
| думать, что constructors вызываются из `main` | глобальные constructors вызываются до `main` runtime-ом |

# 13. Что должно остаться в голове

После этого дня ты должен уметь:

- объяснить путь `source -> object -> executable -> process`;
- понимать, что `main` вызывает runtime;
- знать роль `_start` и `__libc_start_main` на уровне идеи;
- понимать, что такое initial stack, `argc`, `argv`, `envp`;
- объяснить `.init_array` и `.fini_array`;
- использовать `objdump` для просмотра секций и кода;
- не пугаться служебного кода в дизассемблировании.

Если можешь объяснить, почему твой `main` не первая инструкция, день усвоен.
