# Паттерн: libc и 16-byte stack alignment

## Когда нужен

Когда в условии написано:

```text
Запрещается использовать io.inc.
При вызове всех библиотечных функций стек должен быть выровнен по 16 байт.
```

Это относится к вызовам:

```text
printf, scanf, fopen, fclose, fscanf, fprintf, strcmp, strstr, strlen, ...
```

## Что уже известно

CDECL на IA-32:

- аргументы кладутся в стек справа налево;
- результат обычно в `eax`;
- caller чистит стек после вызова;
- `eax/ecx/edx` caller-saved;
- `ebx/esi/edi/ebp` callee-saved.

Для Spring-04 добавляется ещё одно требование: **перед библиотечным `call` стек должен быть выровнен по 16 байт**.

## Что значит “выровнен”

Адрес кратен 16:

```text
esp % 16 == 0
```

Практически это значит: нельзя просто случайно делать `push` сколько получилось и надеяться, что libc примет.

## Безопасная стратегия курса

Не пытайся каждый раз вручную гадать в `main`. Пиши маленькие wrapper-функции:

```text
main -> wrapper -> libc
```

Wrapper отвечает за:

- порядок аргументов;
- временный padding;
- вызов libc;
- восстановление `esp`;
- сохранение callee-saved регистров, если они использовались.

## Универсальная идея padding

Перед вызовом можно запомнить старый `esp`, выровнять стек вниз, сделать `push` аргументов, вызвать функцию, потом вернуть `esp`.

Shape:

```asm
; save original esp somewhere
; reserve/align stack
; push arguments right-to-left
; call libc_function
; restore esp exactly
```

Главное правило: после wrapper стек должен быть ровно таким же, как до wrapper.

## Почему нельзя просто забыть padding

Если ты добавил padding:

```asm
sub esp, 12
```

то потом обязан убрать именно его. Иначе `ret` возьмёт неправильный return address.

## Мини-шаблон мышления для libc call

Перед каждым вызовом проверь:

```text
1. Какие аргументы?
2. В каком порядке push?
3. Нужны адреса или значения?
4. Сколько байт аргументов?
5. Где alignment?
6. Кто чистит стек?
7. Какие регистры могут испортиться?
```

## Пример: `printf("%d\n", x)`

C-shape:

```c
printf("%d\n", x);
```

Stack-shape:

```text
push x
push fmt
call printf
```

После вызова:

```text
caller removes 8 bytes of arguments
```

Но если задача требует 16-byte alignment, аргументы и padding надо считать вместе.

## Частые ошибки

| Ошибка | Симптом |
|---|---|
| стек не выровнен | локально может работать, в ejudge падать |
| padding добавлен, но не убран | `ret` прыгает в мусор |
| перепутали адрес и значение | `scanf/fscanf` пишет не туда |
| забыли, что libc портит `eax/ecx/edx` | значение исчезло после вызова |
| использовали `ebx` без сохранения | вызывающий код может сломаться |

## Как проверить себя

После каждой wrapper-функции задай вопрос:

```text
Если поставить breakpoint до и после wrapper, esp одинаковый?
```

Если нет — стек сломан.
