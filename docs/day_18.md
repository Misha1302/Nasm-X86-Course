# День 18. Reverse engineering: читаем asm как следы C

## Опора на материалы ВШЭ

## Зачем этот день

Это день-детектив. Мы смотрим на asm и восстанавливаем, какой C-код мог его породить. На экзамене это очень полезный навык.

## Главная мысль

Не надо восстанавливать исходник идеально. Нужно восстановить смысл: аргументы, типы, операции, указатели, return.

## Шпаргалка паттернов

```text
[ebp+8]       -> first argument
[ebp+12]      -> second argument
[ebp-4]       -> local variable
movsx byte    -> signed char
movzx word    -> unsigned short
mov [eax],edx -> write through pointer
final eax     -> return value
```

## Разбор примера

```asm
movsx edx, byte [ebp+12]
mov eax, [ebp+16]
mov [eax], edx
movsx eax, word [ebp+8]
mov edx, [ebp+20]
sub edx, eax
mov eax, edx
```

Читаем по шагам:

- `[ebp+8]` — первый аргумент, берётся как `word` и `movsx`, значит похоже на `short`;
- `[ebp+12]` — второй аргумент, `byte` + `movsx`, значит похоже на `signed char`;
- `[ebp+16]` загружается в `eax`, потом `[eax] = edx`, значит это указатель;
- `[ebp+20]` — четвёртый аргумент;
- результат возвращается в `eax`.

Возможный C:

```cpp
int f(short c, signed char d, int* p, int x) {
    *p = d;
    return x - c;
}
```

## Что важно

Это **один возможный** C-код, а не единственный. Компилятор мог получить такой asm из разных похожих программ.

## Мини-челленджи

1. Где в примере указатель?
2. Почему `movsx byte` намекает на signed char?
3. Что возвращает функция?
4. Почему нельзя гарантировать идеально исходный C++?

<details>
<summary>Ответы / подсказки</summary>

1. `[ebp+16]`, потому что его значение используется как адрес для записи.
2. Потому что байт расширяется со знаком до 32 бит.
3. `x - c` в предложенной интерпретации.
4. Разные исходники могут компилироваться в похожий asm.

</details>
