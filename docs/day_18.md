# День 18. Reverse engineering: читаем asm как следы C

## Опора на материалы ВШЭ

`Slides2026-08.pdf`: обратная задача по CDECL, восстановление типов, аргументов, порядка параметров и объявления функции по ассемблерному фрагменту.

## Зачем этот день

Это день-детектив.

До этого мы учились писать asm. Но на экзамене часто бывает обратная ситуация: тебе дают фрагмент ассемблера и спрашивают, что он делает.

Важно не пытаться восстановить исходный C-код идеально. Это почти невозможно: разные C-программы могут дать очень похожий asm.

Цель другая:

```text
понять смысл функции:
аргументы -> типы -> действия -> return value
```

---

## Главная мысль

Reverse engineering asm — это не гадание. Это набор признаков.

Ты смотришь на:

- откуда читаются аргументы;
- какими размерами читается память;
- есть ли `movsx` или `movzx`;
- используется ли значение как адрес;
- что попадает в `eax` перед `ret`;
- какие переходы образуют условия и циклы.

---

## 1. Базовая карта CDECL

Если функция использует классический фрейм:

```asm
push ebp
mov ebp, esp
```

то аргументы обычно лежат так:

```text
[ebp+8]   first argument
[ebp+12]  second argument
[ebp+16]  third argument
[ebp+20]  fourth argument
```

Локальные переменные:

```text
[ebp-4]
[ebp-8]
[ebp-12]
```

Возвращаемое значение чаще всего в:

```text
eax
```

---

## 2. Шпаргалка паттернов

| ASM pattern | Возможный C-смысл |
|---|---|
| `[ebp+8]` | первый аргумент |
| `[ebp+12]` | второй аргумент |
| `[ebp-4]` | локальная переменная |
| `mov eax, [ebp+8]` | аргумент типа `int` или pointer |
| `movsx eax, byte [...]` | signed `char` -> `int` |
| `movzx eax, byte [...]` | unsigned `char` -> `int` |
| `movsx eax, word [...]` | signed `short` -> `int` |
| `movzx eax, word [...]` | unsigned `short` -> `int` |
| `mov [eax], edx` | запись по указателю |
| `lea eax, [...]` | вычисление адреса |
| `call f` | вызов функции |
| финальное значение в `eax` | return value |

Это не строгие доказательства, но очень сильные подсказки.

---

## 3. Как понять, что аргумент — указатель

Если значение аргумента используется как адрес, это почти наверняка pointer.

Пример:

```asm
mov eax, [ebp+8]
mov edx, [eax]
```

Здесь `[ebp+8]` загружается в `eax`, а потом используется `[eax]`.

Возможный C-смысл:

```cpp
int* p;
int value = *p;
```

Ещё пример:

```asm
mov eax, [ebp+8]
mov [eax], edx
```

Возможный C-смысл:

```cpp
*p = edx;
```

Если аргумент используется как база массива:

```asm
mov eax, [ebp+8]
mov edx, [ebp+12]
mov eax, [eax + 4*edx]
```

то `[ebp+8]` может быть `int* a`, а `[ebp+12]` — индекс `i`.

---

## 4. Как понять маленькие типы

Если аргумент или память читается через `byte` или `word`, это намёк на маленький тип.

```asm
movsx eax, byte [ebp+8]
```

Возможный тип:

```cpp
signed char
```

```asm
movzx eax, byte [ebp+8]
```

Возможный тип:

```cpp
unsigned char
```

```asm
movsx eax, word [ebp+8]
```

Возможный тип:

```cpp
short
```

```asm
movzx eax, word [ebp+8]
```

Возможный тип:

```cpp
unsigned short
```

Важно: в CDECL маленькие integer-типы часто передаются на стеке всё равно как 4-байтовые слоты, но код может читать нужную часть или расширять её.

## 5. Разбор главного примера

Фрагмент:

```asm
movsx edx, byte [ebp+12]
mov eax, [ebp+16]
mov [eax], edx
movsx eax, word [ebp+8]
mov edx, [ebp+20]
sub edx, eax
mov eax, edx
```

Разбираем медленно.

### Строка 1

```asm
movsx edx, byte [ebp+12]
```

`[ebp+12]` — второй аргумент.

Читаем `byte` и расширяем со знаком.

Значит, второй аргумент похож на:

```cpp
signed char d
```

Результат кладётся в `edx`.

### Строки 2–3

```asm
mov eax, [ebp+16]
mov [eax], edx
```

`[ebp+16]` — третий аргумент.

Его значение кладут в `eax`, а потом используют как адрес для записи.

Значит, третий аргумент похож на указатель:

```cpp
int* p
```

И действие:

```cpp
*p = d;
```

### Строка 4

```asm
movsx eax, word [ebp+8]
```

`[ebp+8]` — первый аргумент.

Читаем `word` и расширяем со знаком.

Похоже на:

```cpp
short c
```

### Строки 5–7

```asm
mov edx, [ebp+20]
sub edx, eax
mov eax, edx
```

`[ebp+20]` — четвёртый аргумент, вероятно `int x`.

Вычитаем из него `c`:

```cpp
x - c
```

Кладём в `eax`, значит возвращаем.

### Возможный C-код

```cpp
int f(short c, signed char d, int* p, int x)
{
    *p = d;
    return x - c;
}
```

## 6. Почему это не единственный C-код

Один и тот же asm может соответствовать разным исходникам.

Например:

```asm
mov eax, [ebp+8]
add eax, [ebp+12]
ret
```

Возможные C-варианты:

```cpp
int f(int a, int b) { return a + b; }
```

или:

```cpp
unsigned f(unsigned a, unsigned b) { return a + b; }
```

Для сложения битовый результат одинаков. Без контекста нельзя всегда точно сказать signed или unsigned.

Поэтому правильная формулировка:

> Возможный C-код такой-то.

А не:

> Единственный исходник был вот такой.

## 7. Как читать условия

Фрагмент:

```asm
mov eax, [ebp+8]
cmp eax, [ebp+12]
jle .else
mov eax, 1
jmp .end
.else:
mov eax, 0
.end:
ret
```

Читаем:

```asm
cmp eax, [ebp+12]
jle .else
```

Если `a <= b`, идём в else.

Если не идём, значит `a > b`, возвращаем 1.

Возможный C:

```cpp
int f(int a, int b)
{
    if (a > b)
        return 1;
    else
        return 0;
}
```

Или проще:

```cpp
return a > b;
```

## 8. Как читать цикл

Фрагмент:

```asm
xor eax, eax
xor ecx, ecx
.loop:
cmp ecx, [ebp+8]
jge .end
add eax, ecx
inc ecx
jmp .loop
.end:
ret
```

Роли:

```text
eax = sum
ecx = i
[ebp+8] = n
```

Цикл:

```cpp
int sum = 0;
for (int i = 0; i < n; ++i)
    sum += i;
return sum;
```

Признаки:

- `xor ecx, ecx` — `i = 0`;
- `cmp ecx, [ebp+8]`; `jge .end` — пока `i < n`;
- `inc ecx` — `++i`;
- `add eax, ecx` — накопление суммы.

---

## 9. Как читать массивы

Фрагмент:

```asm
mov edx, [ebp+8]
mov ecx, [ebp+12]
mov eax, [edx + 4*ecx]
ret
```

`[ebp+8]` используется как база адреса.

`[ebp+12]` используется как индекс.

`4*ecx` значит элемент размером 4 байта, вероятно `int`.

Возможный C:

```cpp
int f(int* a, int i)
{
    return a[i];
}
```

Если было бы:

```asm
movzx eax, byte [edx + ecx]
```

то возможный C:

```cpp
unsigned char* a;
return a[i];
```

## 10. Порядок анализа asm-фрагмента

Когда видишь неизвестный asm, не читай хаотично.

Иди по плану:

1. Есть ли `push ebp; mov ebp, esp`?
2. Где читаются `[ebp+8]`, `[ebp+12]`, ...?
3. Где есть `[ebp-4]`, `[ebp-8]`?
4. Какие размеры чтения: `byte`, `word`, `dword`?
5. Есть ли `movsx/movzx`?
6. Используется ли значение как адрес?
7. Есть ли `cmp/test/jcc`?
8. Есть ли циклический `jmp` назад?
9. Что лежит в `eax` перед `ret`?
10. Можно ли записать простой C-псевдокод?

## 11. Мини-челленджи

#### 1. Где указатель?

```asm
mov eax, [ebp+12]
mov [eax], edx
```

<details>
<summary>Ответ</summary>

Второй аргумент `[ebp+12]` похож на указатель, потому что его значение используется как адрес для записи.

</details>

#### 2. Что значит `movsx byte`?

```asm
movsx eax, byte [ebp+8]
```

<details>
<summary>Ответ</summary>

Первый аргумент читается как signed byte, то есть возможный тип `signed char`.

</details>

#### 3. Восстанови C

```asm
mov eax, [ebp+8]
imul eax, eax
ret
```

<details>
<summary>Ответ</summary>

```cpp
int f(int x)
{
    return x * x;
}
```

</details>

#### 4. Восстанови массив

```asm
mov eax, [ebp+8]
mov ecx, [ebp+12]
mov eax, [eax + 4*ecx]
ret
```

<details>
<summary>Ответ</summary>

```cpp
int f(int* a, int i)
{
    return a[i];
}
```

</details>

#### 5. Почему нельзя гарантировать исходный C++?

<details>
<summary>Ответ</summary>

Потому что разные исходники могут компилироваться в один и тот же или очень похожий asm, особенно после оптимизаций.

</details>

## 12. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| пытаться восстановить исходник дословно | обычно можно восстановить только смысл |
| игнорировать размер `byte/word/dword` | потеряешь информацию о типах |
| не замечать `movsx/movzx` | перепутаешь signed/unsigned small types |
| считать любой `[eax]` массивом | это может быть просто указатель |
| забыть, что `eax` — return value | потеряешь результат функции |
| читать `[ebp+4]` как аргумент | это return address |
| не отслеживать роли регистров | потеряешь структуру алгоритма |

После этого дня ты должен уметь:

- видеть аргументы по `[ebp+8]`, `[ebp+12]`, ...;

- по `movsx/movzx` предполагать типы;
- отличать значение от указателя;
- понимать, что возвращается через `eax`;
- восстанавливать простой C-псевдокод;
- честно говорить “возможный C-код”, а не “единственный исходник”.

Если ты можешь разобрать функцию на аргументы, действия и return — день усвоен.
