# День 07. Почему для `add` нет `iadd`

## Опора на материалы ВШЭ

`Slides2026-04.pdf`: арифметические команды и знаковая/беззнаковая интерпретация битовых векторов.

## Зачем этот день

Есть `mul/imul`, `div/idiv`, но нет `add/iadd`. Это хороший способ понять, что signed и unsigned — это часто не разные данные, а разные способы прочитать одни и те же биты.

## Главная мысль

`add` работает с битами по модулю `2^w`. Для signed и unsigned сохраняемый битовый результат одинаковый. Разницу потом показывают флаги.

## Пример на 8 битах

```text
  11111111
+ 00000001
-----------
1 00000000
  ^^^^^^^^ stored result
```

Это можно прочитать двумя способами:

```text
unsigned: 255 + 1 = 0 mod 256
signed:   -1 + 1 = 0
```

Биты результата одинаковые: `00000000`.

## Команды дня

```asm
mov eax, [a]
add eax, [b]
sub eax, 10
inc eax
dec eax
neg eax
```

`neg eax` делает `eax = -eax`, то есть фактически `0 - eax`.

## Флаги, которые пока нужны

| Flag | Смысл |
|---|---|
| `CF` | carry/borrow для unsigned |
| `OF` | signed overflow |
| `ZF` | результат равен нулю |
| `SF` | знаковый бит результата |

## Почему для умножения/деления есть signed/unsigned версии

Для сложения и вычитания битовый результат совпадает. Но деление сильно зависит от того, считаем ли число отрицательным.

Например:

```text
0xFFFFFFFF as unsigned = 4294967295
0xFFFFFFFF as signed   = -1
```

Делить эти значения нужно по-разному. Поэтому есть `div/idiv`.

## Мини-челленджи

1. Посчитай в 8 битах `127 + 1`.
2. Посчитай в 8 битах `255 + 1`.
3. Объясни своими словами, почему нет `iadd`.
4. Что меняет `add eax, ebx`?

<details>
<summary>Ответы / подсказки</summary>

1. Битовый результат `10000000`; как signed это `-128`, значит signed overflow.
2. Битовый результат `00000000`; был unsigned carry.
3. Потому что сохраняемый битовый результат сложения одинаков для signed/unsigned.
4. Меняет `eax`, выставляет флаги.

</details>
