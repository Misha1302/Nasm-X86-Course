# День 07. Почему для `add` нет `iadd`

## Опора на материалы ВШЭ

`Slides2026-04.pdf`: арифметические команды, знаковая и беззнаковая интерпретация битовых векторов, флаги результата.

## Зачем этот день

В x86 есть пары:

```text
mul  / imul
div  / idiv
```

То есть unsigned-умножение и signed-умножение, unsigned-деление и signed-деление.

Но нет пары:

```text
add / iadd
sub / isub
```

На первый взгляд странно. В C++ же есть `int` и `unsigned int`. Почему процессору не нужны разные команды сложения?

Сегодня через этот вопрос мы разберём главное: **signed и unsigned — это часто не разные данные, а разные способы читать одни и те же биты**.

---

## Главная мысль

`add` складывает битовые векторы по модулю `2^w`, где `w` — размер операнда.

Для signed и unsigned сохраняемый битовый результат одинаковый.

Разница появляется в интерпретации:

- unsigned-смысл смотрит на `CF`;
- signed-смысл смотрит на `OF`.

---

## Самый короткий пример

Возьмём 8 бит.

```text
  11111111
+ 00000001
-----------
1 00000000
  ^^^^^^^^ stored result
```

В 8-битный результат сохраняется только:

```text
00000000
```

Теперь читаем это двумя способами.

### Как unsigned

```text
11111111 = 255
00000001 = 1
255 + 1 = 256
256 mod 256 = 0
```

Результат `0`, был перенос за пределы 8 бит. Это важно для `CF`.

### Как signed

```text
11111111 = -1
00000001 = 1
-1 + 1 = 0
```

Результат тоже `0`, и signed overflow здесь нет.

Биты результата одинаковые. Поэтому отдельная команда `iadd` не нужна.

---

## Что делает `add`

```asm
add eax, ebx
```

Это значит:

```cpp
eax = eax + ebx;
```

Важно: результат пишется в первый операнд.

До:

```text
eax = 10
ebx = 3
```

После:

```text
eax = 13
ebx = 3
```

`ebx` не меняется.

---

## Команды дня

| Команда | Пример | C++-смысл |
|---|---|---|
| `add` | `add eax, ebx` | `eax += ebx` |
| `sub` | `sub eax, 10` | `eax -= 10` |
| `inc` | `inc eax` | `eax++` |
| `dec` | `dec eax` | `eax--` |
| `neg` | `neg eax` | `eax = -eax` |

Пример:

```asm
mov eax, [a]
add eax, [b]
sub eax, 10
inc eax
dec eax
neg eax
```

Последняя команда:

```asm
neg eax
```

делает:

```text
eax = 0 - eax
```

---

## `inc` и `dec`

```asm
inc eax
```

почти как:

```asm
add eax, 1
```

```asm
dec eax
```

почти как:

```asm
sub eax, 1
```

Но есть тонкость: `inc/dec` не меняют `CF`, а `add/sub` меняют. Для первых задач это редко критично, но на уровне флагов это важно помнить.

Если не уверен, используй `add/sub`: они проще для рассуждений о флагах.

---

## `neg`: как получить минус

```asm
neg eax
```

Если:

```text
eax = 5
```

после `neg`:

```text
eax = -5
```

На уровне битов это two's complement:

```text
-x = ~x + 1
```

Например в 8 битах:

```text
5      = 00000101
~5     = 11111010
~5 + 1 = 11111011 = -5
```

---

## Флаги: пока только главное

После арифметики процессор обновляет флаги.

| Flag | Смысл |
|---|---|
| `CF` | carry/borrow для unsigned |
| `OF` | signed overflow |
| `ZF` | результат равен нулю |
| `SF` | старший бит результата |

### `CF`: unsigned-перенос

8-битно:

```text
255 + 1 -> 0, CF = 1
```

Почему? Потому что результат не поместился в 8 бит как unsigned.

### `OF`: signed overflow

8-битный signed диапазон:

```text
-128..127
```

```text
127 + 1 -> -128 по битам, OF = 1
```

Битовый результат есть, но как signed-математика он неправильный.

---

## Почему для деления есть `div/idiv`

Для сложения битовый результат совпадает.

Но для деления уже нет.

Один и тот же набор битов:

```text
0xFFFFFFFF
```

может означать:

```text
unsigned: 4294967295
signed:   -1
```

Теперь делим на 3:

```text
4294967295 / 3 = 1431655765
-1 / 3 = 0    ; в C-подобном целочисленном делении к нулю
```

Это совсем разные результаты. Поэтому у деления есть разные команды:

```asm
div   ; unsigned
idiv  ; signed
```

Умножение тоже имеет signed/unsigned формы, особенно когда нужен полный расширенный результат.

## Маленькая программа: `a - b`

```asm
section .data
    fmtIn db "%d", 0
    fmtOut db "%d", 10, 0

section .bss
    a resd 1
    b resd 1

section .text
    extern scanf
    extern printf
    global main

main:
    push a
    push fmtIn
    call scanf
    add esp, 8

    push b
    push fmtIn
    call scanf
    add esp, 8

    mov eax, [a]
    sub eax, [b]

    push eax
    push fmtOut
    call printf
    add esp, 8

    xor eax, eax
    ret
```

Главные строки:

```asm
mov eax, [a]
sub eax, [b]
```

Это:

```cpp
answer = a - b;
```

## Мини-челленджи

### 1. `127 + 1` в 8 битах

Что получится по битам? Что с signed-смыслом?

<details>
<summary>Ответ</summary>

```text
01111111 + 00000001 = 10000000
```

По signed 8-bit это `-128`, хотя математически было `128`. Значит, signed overflow, `OF = 1`.

</details>

### 2. `255 + 1` в 8 битах

Что получится?

<details>
<summary>Ответ</summary>

```text
11111111 + 00000001 = 1 00000000
```

Сохраняется `00000000`, есть unsigned carry, `CF = 1`.

</details>

### 3. Почему нет `iadd`?

Скажи одной фразой.

<details>
<summary>Ответ</summary>

Потому что сохраняемый битовый результат сложения одинаков для signed и unsigned; отличается только интерпретация флагов.

</details>

### 4. Что меняет `add eax, ebx`?

<details>
<summary>Ответ</summary>

Меняет `eax` и флаги. `ebx` не меняется.

</details>

### 5. Что делает `neg eax`?

<details>
<summary>Ответ</summary>

Делает `eax = -eax`, то есть `0 - eax`.

</details>

## Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| думать, что signed и unsigned — разные биты | это разные интерпретации тех же битов |
| путать `CF` и `OF` | `CF` для unsigned, `OF` для signed |
| читать `add eax, ebx` как чистое выражение | это `eax += ebx`, `eax` меняется |
| забывать, что `inc/dec` имеют особенность с `CF` | иногда лучше использовать `add/sub` |
| считать `neg` магическим | это two's complement: `~x + 1` |
