## Главная мысль

Файл `.asm` сам не запускается.

Сначала его собирает NASM в object file. Потом linker собирает executable. Потом ОС загружает executable в память и запускает процесс.

```text
main.asm
   |
   | nasm -f elf32
   v
main.o
   |
   | gcc -m32 -no-pie
   v
main
   |
   | ./main
   v
output
```

SASM обычно делает эти шаги за тебя. Мы хотим понимать их вручную.

---

## Минимальная NASM-программа

Создай файл `main.asm`:

```asm
section .text
global main

main:
    xor eax, eax
    ret
```

Что здесь происходит:

| Строка | Смысл |
|---|---|
| `section .text` | дальше идёт код |
| `global main` | метка `main` видна линкеру |
| `main:` | начало функции `main` |
| `xor eax, eax` | сделать `eax = 0` |
| `ret` | вернуться из функции |

Почему `xor eax, eax`, а не `mov eax, 0`?

Оба варианта дают ноль, но `xor reg, reg` — частый короткий идиоматический способ обнуления регистра. В этом курсе ты будешь видеть его постоянно.

## Сборка из терминала

Команды:

```bash
nasm -f elf32 main.asm -o main.o
gcc -m32 -no-pie main.o -o main
./main
echo $?
```

Разберём каждую.

### `nasm -f elf32 main.asm -o main.o`

NASM читает текстовый asm и делает object file.

```text
main.asm  ->  main.o
```

`-f elf32` значит: создать 32-битный ELF object file. Это важно, потому что мы работаем с IA-32.

### `gcc -m32 -no-pie main.o -o main`

`gcc` здесь используется как удобный linker driver.

Он:

- вызывает линковщик;
- подключает стартовый код C runtime;
- подключает libc, если нужны `printf`, `scanf` и т.п.;
- делает итоговый executable.

`-m32` говорит: линковать как 32-битную программу.

## Важное уточнение: `-no-pie`

Почему не просто `gcc -m32 main.o -o main`?

Многие современные Linux-дистрибутивы по умолчанию собирают PIE-executable: position-independent executable.

А первые NASM-программы курса используют простые абсолютные адреса меток:

```asm
push fmtOut
mov eax, [x]
jmp [.table + 4*eax]
```

Для такого учебного IA-32 кода проще и честнее собирать без PIE:

```bash
gcc -m32 -no-pie main.o -o main
```

PIE, GOT, PLT и position-independent code — отдельная тема. Мы её упомянем позже, но не будем мешать в первые задачи.

---

### `./main`

Запуск программы.

### `echo $?`

Показать код возврата последней программы.

Наша программа возвращает `0`, потому что перед `ret` мы сделали:

```asm
xor eax, eax
```

В C это примерно как:

```cpp
return 0;
```

## Что такое `.asm`, `.o`, executable

| Файл | Что это |
|---|---|
| `main.asm` | текст на ассемблере |
| `main.o` | object file: машинный код и данные, но ещё не финальная программа |
| `main` | executable: файл, который ОС может загрузить и запустить |

Object file часто содержит:

- машинный код;
- данные;
- таблицы символов;
- relocation-записи;
- ссылки на внешние функции.

То есть `.o` уже не текст, но ещё не полноценная программа.

---

## Почему не просто `ld`

Можно линковать напрямую через `ld`, но для начала это лишняя боль. Когда в программе есть `main`, `printf`, `scanf`, стартовый код и libc, проще использовать:

```bash
gcc -m32 -no-pie main.o -o main
```

`gcc` сам правильно подставит нужные системные куски.

Для учебного курса это нормально: мы изучаем ассемблер, а не мучаемся с низкоуровневой линковкой в первый же день.

## `objdump`: смотрим внутрь

Когда программа собралась, полезно посмотреть, что внутри.

```bash
objdump -d -M intel main
```

- `-d` — дизассемблировать код;
- `-M intel` — показывать Intel-синтаксис, похожий на NASM.

Можно увидеть не только твой `main`, но и служебный код вокруг него. Это нормально.

Посмотреть секции:

```bash
objdump -h main
```

Посмотреть содержимое `.data`:

```bash
objdump -s -j .data main
```

## Что делает SASM

Когда ты нажимаешь Run в SASM, он примерно делает то же самое:

```text
SASM Run
   |
   v
save .asm
   |
   v
nasm
   |
   v
linker / gcc
   |
   v
run executable
   |
   v
show output
```

То есть SASM — не отдельный язык. Это удобная оболочка вокруг NASM, линковщика и запуска.

## Частая боль: нет 32-битных библиотек

Если `gcc -m32` ругается, часто проблема не в твоём asm, а в системе: не хватает 32-битных библиотек.

Ubuntu/Debian:

```bash
sudo apt install nasm gcc gcc-multilib libc6-dev-i386
```

Fedora:

```bash
sudo dnf install nasm gcc glibc-devel.i686 libgcc.i686
```

Если после установки всё равно не собирается, сначала проверь простую C-программу:

```c
int main() { return 0; }
```

Сборка:

```bash
gcc -m32 test.c -o test
```

Если даже это падает — проблема точно в окружении, а не в NASM-коде.

## Мини-практика: первая программа

Сделай файл `main.asm`, собери и запусти:

```asm
section .text
global main

main:
    mov eax, 7
    ret
```

Потом:

```bash
nasm -f elf32 main.asm -o main.o
gcc -m32 -no-pie main.o -o main
./main
echo $?
```

Ожидаемый код возврата:

```text
7
```

Почему? Потому что значение в `eax` считается возвращаемым значением `main`.

Важно: код возврата процесса обычно должен быть маленьким неотрицательным числом. Для больших вычислений так выводить результат не надо — позже будем использовать `printf`.

## Мини-челленджи

### 1. Разница файлов

Объясни разницу:

```text
main.asm
main.o
main
```

<details>
<summary>Ответ</summary>

`main.asm` — текстовый исходник.

`main.o` — object file после NASM.

`main` — executable после линковки.

</details>

### 2. Найди `main`

Собери программу и выполни:

```bash
objdump -d -M intel main | less
```

Найди метку `<main>`.

<details>
<summary>Подсказка</summary>

В `less` можно нажать `/`, набрать `main`, нажать Enter.

</details>

### 3. Почему нужен `-m32`?

Ответь одной фразой.

<details>
<summary>Ответ</summary>

Потому что мы собираем 32-битный IA-32 object file и хотим линковать его как 32-битную программу.

</details>

## Типовые ошибки

| Ошибка | Что делать |
|---|---|
| `file format not recognized` | проверить `nasm -f elf32` |
| ошибки при `gcc -m32` | поставить 32-битные библиотеки |
| `undefined reference to main` | проверить `global main` и метку `main:` |
| не найден `nasm` | установить NASM |
| в `objdump` много лишнего кода | это runtime/startup, ищи `<main>` |
