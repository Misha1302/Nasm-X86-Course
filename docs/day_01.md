## Зачем этот день

Ты уже умеешь писать C++. Например:

```cpp
int x = 2;
int y = 3;
int z = x + y;
```

Для тебя это одна понятная строка: “сложить `x` и `y`, положить в `z`”.

Но процессор не видит “переменные C++” и не понимает выражение целиком. Он выполняет маленькие команды: прочитать значение, положить в регистр, сложить, записать, перейти по адресу.

Сегодня не нужно писать сложный NASM. Задача дня — понять, **где ассемблер находится между C++ и железом**.

---

## Главная мысль

C++ — удобный язык для человека.

Ассемблер — почти прямой текстовый вид команд процессора.

Машинный код — байты, которые CPU реально исполняет.

```text
C++ source
   |
   | compiler
   v
Assembly source
   |
   | assembler
   v
Object file (.o)
   |
   | linker
   v
Executable
   |
   | OS loader
   v
Process in memory
   |
   | CPU executes machine instructions
   v
Result
```

Можно думать так:

```text
C++ говорит:       что хотим посчитать.
Assembly говорит:  какими командами это сделать.
Machine code:      как эти команды закодированы байтами.
CPU:               просто выполняет байты.
```

---

## Почему ассемблер вообще нужен

Не для того, чтобы все программы писать вручную. Это почти всегда плохая идея.

Ассемблер нужен, чтобы видеть нижний уровень:

- где реально лежат данные;
- почему происходит переполнение;
- что значит “адрес”;
- как работает вызов функции;
- почему выход за границы массива опасен;
- что делает компилятор;
- как читать дизассемблирование на экзамене или в отладке.

Хорошая формулировка:

> Ассемблер — это способ перестать гадать, что делает машина.

---

## Где asm реально используется

Если очень абстрактно, ассемблер нужен в трёх случаях:

| Причина | Что это значит |
|---|---|
| Привязка к архитектуре | нужна конкретная инструкция конкретного процессора |
| Производительность | нужно ускорить очень горячий участок |
| Полный контроль | нужно управлять регистрами, стеком, ABI, адресами |

Более конкретно:

- операционные системы, драйверы, загрузчики;
- компиляторы, интерпретаторы, JIT;
- runtime-библиотеки языков;
- libc и низкоуровневые функции памяти/строк;
- кодеки, криптография, обработка изображений;
- HFT и low-latency системы;
- reverse engineering и security;
- embedded и микроконтроллеры.

Но важно не романтизировать:

> Обычно на asm пишут не всю программу, а маленькие критические куски.

Современный компилятор часто генерирует очень хороший код. Хороший low-level программист сначала измеряет, смотрит дизассемблирование, понимает bottleneck, и только потом решает, нужен ли ручной asm.

## Аккуратные примеры для рассказа

Эти примеры можно использовать, но публично лучше проверять точные факты под текущую версию проекта.

| Пример | Как говорить аккуратно |
|---|---|
| FFmpeg | В медиабиблиотеках часто есть архитектурно-зависимые SIMD/asm-оптимизации в горячих местах. Не надо говорить, что “весь FFmpeg на asm”. |
| KolibriOS | Есть маленькие ОС, где ассемблер используется очень широко; KolibriOS/FASM часто приводят как пример. Размер и формулировку лучше проверить. |
| .NET Runtime | В рантаймах языков есть платформенно-зависимые низкоуровневые куски: startup, interop, atomics, helpers, memory routines. Конкретные утверждения про строки лучше проверять. |
| libc | Реализации libc могут иметь оптимизированные версии `memcpy`, `memmove`, `strlen` и похожих функций под конкретные архитектуры. |

Правильный стиль: не “я слышал байку”, а “в таких проектах действительно бывают архитектурно-зависимые оптимизации; конкретику проверяем”.

## Первый мостик: C++ → машинные шаги

C++:

```cpp
z = x + y;
```

Процессор не решает это выражение “целиком”. Ему нужно что-то вроде:

```asm
mov eax, [x]    ; взять x из памяти в регистр eax
add eax, [y]    ; прибавить y к eax
mov [z], eax    ; записать результат обратно в память
```

Пока не надо глубоко понимать квадратные скобки. Это будет в дне 05.

Сейчас важно ощущение:

```text
C++ expression
   |
   v
load data -> compute in register -> store result
```

То есть выражение распадается на маленькие шаги.

## Почему процессор не знает `while`

В C++:

```cpp
while (x > 0) {
    x--;
}
```

А на низком уровне это идея:

```text
.loop:
    проверить x > 0
    если нет — уйти в .end
    уменьшить x
    прыгнуть обратно в .loop
.end:
```

То есть `while` — это удобная форма для человека. Для процессора это сравнение, условный переход и безусловный переход.

В этом курсе почти всё будет раскладываться так:

| C++ | Низкоуровневая идея |
|---|---|
| переменная | значение в памяти или регистре |
| выражение | последовательность маленьких операций |
| `if` | сравнение + условный переход |
| `while` | метка + проверка + переход назад |
| функция | `call`, стек, `ret` |
| массив | базовый адрес + индекс * размер элемента |

## Почему мы пока не берём x64

x64 важен, но для первых недель он будет мешать.

В x64:

- больше регистров;
- другие имена: `rax`, `rbx`, `rcx`, ...;
- аргументы часто передаются через регистры;
- соглашение вызова зависит от ОС;
- стек всё ещё важен, но выглядит иначе.

А курс и задачи сейчас про `nasm-x86`, то есть IA-32:

```text
eax, ebx, ecx, edx
esp, ebp
stack arguments
cdecl
```

Поэтому договор простой:

> x64 существует, но пока не смешиваем две модели в голове.
