# День 01. Что вообще происходит, когда программа запускается

## Опора на материалы ВШЭ

`Slides2026-00 - intro.pdf`, `Slides2026-01.pdf`: зачем курс связывает C, ассемблер, архитектуру, производительность и безопасность.

## Зачем этот день

Ты уже умеешь писать C++. Но процессор не исполняет C++ напрямую. Он исполняет простые команды: взять число, положить в регистр, сложить, записать, перейти по адресу.

Ассемблер нужен не для того, чтобы всё писать вручную. Он нужен, чтобы перестать бояться нижнего уровня: переполнений, указателей, падений, strange behavior, дизассемблирования и вопросов на экзамене.

## Главная мысль

C++ — удобный язык для человека. Ассемблер — почти прямой текстовый вид команд процессора.

```text
C++ source
   |
   | compiler
   v
Assembly source
   |
   | assembler
   v
Object file
   |
   | linker
   v
Executable
   |
   | OS loader
   v
Process in memory
   |
   | CPU executes machine instructions
   v
Result
```

## Первый мостик: C++ → машинные шаги

В C++ ты пишешь:

```cpp
z = x + y;
```

Процессор не видит выражение целиком. Ему нужно примерно такое:

```asm
mov eax, [x]    ; взять x из памяти в регистр
add eax, [y]    ; прибавить y
mov [z], eax    ; сохранить результат обратно в память
```

Пока не нужно идеально понимать квадратные скобки. Запомни ощущение: **всё распадается на маленькие шаги**.

## Где asm реально полезен

Если совсем абстрактно, ассемблер появляется в трёх случаях:

1. нужна привязка к конкретной архитектуре;
2. важна производительность в горячем участке;
3. нужен полный контроль, который обычный язык обычно прячет.

Более конкретно:

- ОС, драйверы, загрузчики;
- компиляторы, интерпретаторы, JIT;
- runtime-библиотеки языков;
- высоконагруженные библиотеки, кодеки, криптография;
- reverse engineering и security;
- embedded и низкоуровневая отладка.

Но важная мысль: **обычно на asm пишут не всю программу, а маленькие критические куски**.

Например, в разговоре часто вспоминают FFmpeg, KolibriOS, runtime-библиотеки, libc и архитектурно-оптимизированные функции. Эти примеры полезны, но если ты используешь их в докладе или публичном тексте, лучше проверить точные факты: что именно написано на asm, в какой версии, на какой архитектуре и насколько большая там доля такого кода.

## Мини-челленджи

1. Разложи `a = b + c - 7` на шаги `load/add/sub/store`.
2. Почему процессор не знает слово `while`?
3. Что ближе к реальности: “переменная живёт сама по себе” или “значение лежит в памяти/регистре”?

<details>
<summary>Ответы / подсказки</summary>

1. Примерно: загрузить `b`, прибавить `c`, вычесть `7`, сохранить в `a`.
2. `while` компилятор превращает в сравнения, условные переходы и метки.
3. На низком уровне есть байты памяти, регистры и инструкции. “Переменная” — удобная абстракция языка.

</details>

## Что должно остаться в голове

- asm — не “древний C++”, а язык инструкций процессора;
- C++-выражение превращается в загрузки, операции и сохранения;
- дальше мы будем учиться видеть регистры, память, стек и флаги.
