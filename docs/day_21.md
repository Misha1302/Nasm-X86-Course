## Зачем этот день

В C++ ошибка памяти часто выглядит невинно:

```cpp
int a[3];
a[10] = 123;
```

Кажется: ну просто взяли “не тот элемент”.

Но на уровне asm и памяти элемента `a[10]` не существует. Есть только адрес, который оказался за пределами массива. По этому адресу может лежать что угодно:

- другая локальная переменная;
- сохранённый `ebp`;
- адрес возврата;
- данные другого объекта;
- служебная информация runtime.

Сегодня мы разбираем, почему ошибки памяти опасны и какие защиты пытаются их остановить.

## Главная мысль

В памяти рядом могут лежать и данные, и служебные значения управления программой.

Если программа пишет за пределы буфера, она может испортить не только “чужие данные”, но и информацию, от которой зависит управление выполнением.

```text
ошибка индекса -> запись не туда -> повреждение памяти -> странное поведение / crash / security bug
```

# 1. Почему `a[10]` опасен

C++:

```cpp
int a[3];
a[10] = 123;
```

Если `a` лежит на стеке, рядом с ним могут быть другие данные фрейма.

Упрощённая картинка:

```text
higher addresses
+------------------+
| return address   |
+------------------+
| saved ebp        |
+------------------+
| local variable   |
+------------------+
| a[2]             |
+------------------+
| a[1]             |
+------------------+
| a[0]             |
+------------------+
lower addresses
```

`a[0]`, `a[1]`, `a[2]` — нормальные элементы.

`a[3]`, `a[4]`, ... — это уже не элементы массива. Это просто память рядом.

Что именно там окажется, зависит от компилятора, оптимизаций, ABI и защит.

# 2. Undefined behavior

В C/C++ выход за границы массива — undefined behavior.

Это значит не “программа обязана упасть”.

Это значит:

```text
язык больше не обещает никакого нормального поведения.
```

Возможные варианты:

- программа работает “как будто нормально”;
- печатает мусор;
- падает;
- ломает соседние переменные;
- работает по-разному на разных компиляторах;
- ломается только с оптимизациями;
- создаёт уязвимость.

Самое неприятное: если сегодня “работает”, это не значит, что код правильный.

# 3. Buffer overflow на уровне идеи

Buffer overflow — это запись за пределы буфера.

Пример:

```c
char buf[16];
// слишком длинный ввод записался в buf
```

Упрощённый стек:

```text
higher addresses
+------------------+
| return address   |
+------------------+
| saved ebp        |
+------------------+
| local buffer     |
| buf[0..15]       |
+------------------+
lower addresses
```

Если запись выходит за `buf`, она может дойти до сохранённого `ebp` или address return.

Почему это страшно?

`ret` берёт адрес возврата со стека. Если адрес возврата испорчен, управление может уйти не туда.

---

# 4. Stack canary

Stack canary — защитное значение между локальным буфером и служебными данными фрейма.

```text
higher addresses
+------------------+
| return address   |
+------------------+
| saved ebp        |
+------------------+
| canary           |
+------------------+
| local buffer     |
+------------------+
lower addresses
```

Идея:

1. при входе в функцию положить canary в стек;
2. перед выходом проверить, не изменился ли он;
3. если изменился — вероятно, буфер переписал память выше себя;
4. аварийно завершить программу.

Псевдологика:

```text
on entry:
    stack_canary = secret_value

before return:
    if stack_canary != secret_value:
        abort
```

Canary не “исправляет” ошибку. Он пытается её обнаружить до `ret`.

---

# 5. ASLR

ASLR = Address Space Layout Randomization.

Идея: каждый запуск раскладывать важные области памяти по разным адресам.

Без ASLR:

```text
library address = same every run
stack address   = same every run
```

С ASLR:

```text
library address = changes
stack address   = changes
heap address    = changes
```

Это усложняет атаки, которые зависят от знания точных адресов.

Но ASLR не исправляет сам bug. Если в программе есть выход за границы, он всё ещё есть.

---

# 6. DEP / NX

DEP/NX запрещает исполнять страницы памяти, предназначенные для данных.

Идея:

```text
code pages: executable
data/stack/heap pages: not executable
```

Если кто-то пытается выполнить байты из стека как код, процессор/ОС должны запретить это.

DEP/NX отвечает на идею:

> данные не должны становиться кодом.

Но это тоже не исправляет buffer overflow. Это только усложняет превращение ошибки в выполнение произвольного кода.

---

# 7. FORTIFY_SOURCE

`FORTIFY_SOURCE` — набор дополнительных проверок для некоторых функций libc, когда компилятор может понять размер буфера.

Например, для опасных операций копирования строк/памяти runtime может добавить проверку:

```text
не копируем ли мы больше, чем размер буфера?
```

Это не универсальная защита от всех ошибок памяти. Но она может поймать часть типичных ошибок раньше.

---

# 8. CFI / CET на уровне идеи

CFI = Control-Flow Integrity.

Идея: программа должна переходить только туда, куда разрешено её нормальной структурой.

CET — аппаратные механизмы Intel для усиления защиты control flow, например shadow stack и indirect branch tracking.

Для курса достаточно понимать:

```text
canary -> пытается заметить порчу стека;
ASLR   -> скрывает точные адреса;
DEP/NX -> запрещает исполнять данные;
CFI/CET -> ограничивает допустимые переходы управления.
```

---

# 9. Return-to-libc на уровне идеи

DEP/NX запрещает исполнять код на стеке. Но в процессе уже есть готовый код библиотек.

Return-to-libc — историческая идея атаки: вместо выполнения кода из стека попытаться передать управление уже существующей функции библиотеки.

Для курса важно только это:

```text
защиты закрывают одни классы атак,
но ошибки памяти всё равно опасны,
потому что управление программой можно пытаться перенаправлять разными способами.
```

Не нужно изучать практические цепочки вызовов. Нужна концептуальная связь: стек, return address, библиотеки, control flow.

---

# 10. Почему это важно обычному C++-программисту

Потому что многие “странные баги” становятся понятными:

```cpp
int a[3];
int x = 10;
a[3] = 999;
```

Если `x` лежал рядом, он мог внезапно измениться.

Или:

```cpp
char buf[8];
scanf("%s", buf);
```

Если ввод длиннее 7 символов плюс ноль-терминатор, запись выйдет за пределы буфера.

Ассемблер помогает не магически “защититься”, а увидеть реальную причину: адреса, стек, соседние данные.

# 11. Мини-челленджи

### 1. Что может лежать рядом с локальным буфером?

<details>
<summary>Ответ</summary>

Другие локальные переменные, canary, сохранённый `ebp`, адрес возврата, временные значения. Точный layout зависит от компилятора и настроек.

</details>

### 2. Что проверяет canary?

<details>
<summary>Ответ</summary>

Не изменилось ли специальное защитное значение между буфером и служебными данными фрейма.

</details>

### 3. Чем ASLR отличается от DEP/NX?

<details>
<summary>Ответ</summary>

ASLR рандомизирует адреса. DEP/NX запрещает исполнять память данных как код.

</details>

### 4. Почему `a[10]` при `a[3]` — не “просто другой элемент”?

<details>
<summary>Ответ</summary>

Потому что такого элемента массива нет. Это обращение к чужой памяти за пределами объекта.

</details>

### 5. Исправляет ли canary ошибку переполнения?

<details>
<summary>Ответ</summary>

Нет. Canary только пытается обнаружить повреждение перед возвратом из функции.

</details>

# 12. Типовые ошибки понимания

| Ошибка | Почему плохо |
|---|---|
| думать, что UB обязан приводить к crash | UB может выглядеть как “нормальная работа” |
| считать canary полной защитой | он обнаруживает часть переполнений, но не исправляет bug |
| путать ASLR и DEP/NX | одно про адреса, другое про исполнение памяти |
| думать, что если есть защиты, ошибки памяти не важны | защиты не отменяют ошибку |
| забывать, что layout зависит от компилятора | нельзя всегда угадывать точное соседство переменных |
| воспринимать return-to-libc как практическую цель курса | здесь нужна только концепция для понимания защит |
