# День 21. Ошибки памяти и защиты

## Опора на материалы ВШЭ

`Slides2026-0xE.pdf`: buffer overflow, return-to-libc, canary, ASLR, DEP/NX, FORTIFY_SOURCE, CFI/CET.

## Зачем этот день

`a[10]` при массиве размера 3 — это не “просто не тот элемент”. Это обращение к соседней памяти, где могут лежать служебные данные.

## Главная мысль

Ошибки памяти опасны, потому что данные, указатели и адреса возврата лежат в одной адресуемой памяти. Защиты пытаются это обнаружить или усложнить.

## Stack frame with canary

```text
higher addresses
+------------------+
| return address   |
+------------------+
| saved ebp        |
+------------------+
| canary           |
+------------------+
| local buffer     |
+------------------+
lower addresses
```

Перед возвратом функция проверяет, не изменился ли canary. Если изменился — вероятно, было переполнение буфера.

## Пример ошибки

```cpp
int a[3];
a[10] = 123; // запись за пределы массива
```

Что именно сломается, зависит от layout фрейма, компилятора, оптимизаций и защит. Но идея одна: ты пишешь не в элемент массива, а в чужую память.

## Основные защиты

| Defense | Idea |
|---|---|
| Stack canary | обнаружить перезапись фрейма |
| ASLR | рандомизировать адреса |
| DEP/NX | запретить исполнение данных как кода |
| FORTIFY_SOURCE | добавить проверки для части libc-вызовов |
| CFI/CET | ограничить допустимые переходы управления |

## Важная рамка

Мы изучаем это, чтобы понимать ошибки и защиты, а не чтобы писать эксплойты. Для курса достаточно уметь объяснить, почему ошибка опасна и какая защита что делает.

## Мини-челленджи

2. Что проверяет canary?
3. Чем ASLR отличается от DEP/NX?
4. Почему без понимания стека безопасность выглядит магией?

<details>
<summary>Ответы / подсказки</summary>

- Canary проверяет, не была ли перезаписана область между буфером и служебными данными.
- ASLR меняет адреса, DEP/NX запрещает исполнять страницы данных.
