# День 04. Почему `eax`, `ax`, `ah`, `al` — это не четыре переменные

## Опора на материалы ВШЭ

`Slides2026-02.pdf`: регистры IA-32, вложенность регистров и размеры операндов.

## Зачем этот день

В C++ ты обычно думаешь так: одна переменная — одно имя.

```cpp
int x;
char c;
```

В x86 всё чуть хитрее. Один и тот же физический регистр может иметь несколько имён для разных частей.

Самый важный пример:

```text
eax, ax, ah, al
```

Это не четыре независимые переменные. Это части одного регистра.

Если это не понять, потом будут странные ошибки в задачах с байтами, вводом, сдвигами и `movsx/movzx`.

---

## Главная мысль

`eax`, `ax`, `ah`, `al` перекрываются.

```text
31                              16 15       8 7        0
+--------------------------------+----------+----------+
|          high 16 bits          |    AH    |    AL    |
+--------------------------------+----------+----------+
                                  \__________ _________/
                                             |
                                             AX

EAX = full 32-bit register
AX  = low 16 bits of EAX
AH  = high byte of AX
AL  = low byte of AX
```

То есть:

```text
EAX содержит AX,
AX содержит AH и AL.
```

Если меняешь `AL`, меняется младший байт `EAX`.

---

## Пример руками

Пусть:

```text
eax = 0x12345678
```

Тогда:

```text
eax = 0x12345678
ax  =     0x5678
ah  =       0x56
al  =       0x78
```

Почему?

`AX` — младшие 16 бит `EAX`, то есть `0x5678`.

`AH` — старший байт `AX`, то есть `0x56`.

`AL` — младший байт `AX`, то есть `0x78`.

Теперь команда:

```asm
mov al, 0
```

Она меняет только младший байт.

Результат:

```text
eax = 0x12345600
```

Не `0`. Не `0x00000000`. Только младший байт стал нулём.

---

## Ещё один пример

Пусть:

```text
eax = 0xDEADBEEF
```

Тогда:

```text
ax = 0xBEEF
ah = 0xBE
al = 0xEF
```

Если выполнить:

```asm
mov ah, 0x11
```

то `ax` станет:

```text
0x11EF
```

а `eax` станет:

```text
0xDEAD11EF
```

Потому что `ah` — это байт на позициях 8..15.

## Основные регистры IA-32

В курсе чаще всего будут встречаться эти регистры:

| Register | Как о нём думать в этом курсе |
|---|---|
| `eax` | результат, аккумулятор, return value функции |
| `ebx` | обычный рабочий регистр; в CDECL функция должна его сохранять |
| `ecx` | счётчик; для переменных сдвигов используется `cl` |
| `edx` | старшая часть `edx:eax`; остаток после деления |
| `esi` | source/index register, часто указатель или индекс |
| `edi` | destination/index register, часто указатель или индекс |
| `esp` | stack pointer, вершина стека |
| `ebp` | frame pointer, база фрейма функции |
| `eip` | instruction pointer, адрес следующей инструкции |
| `eflags` | флаги результата |

Пока не надо запоминать исторические названия вроде accumulator/base/count/data глубоко. Но полезно знать интуицию:

```text
eax -> часто результат;
ecx -> часто счётчик;
edx -> часто деление;
esp/ebp -> стек и функции.
```

---

## Почему для сдвига нужен `cl`

В x86 команды сдвига могут брать количество сдвига из `cl`, то есть младшего байта `ecx`.

```asm
mov cl, 8
shl eax, cl
```

Не из всего `ecx`, а именно из `cl`.

Поэтому `ecx` часто всплывает в задачах со сдвигами и циклами.

## Размеры операндов

Ассемблеру часто нужно знать, сколько байтов ты хочешь читать или писать.

| Размер | Биты | Байты | Похоже на C/C++ |
|---|---:|---:|---|
| `byte` | 8 | 1 | `char`, `uint8_t` |
| `word` | 16 | 2 | `short`, `uint16_t` |
| `dword` | 32 | 4 | `int`, `uint32_t`, pointer в IA-32 |
| `qword` | 64 | 8 | `long long`, `double` |

Примеры:

```asm
mov al, byte [x]      ; прочитать 1 байт
mov ax, word [x]      ; прочитать 2 байта
mov eax, dword [x]    ; прочитать 4 байта
```

Иногда NASM сам понимает размер по регистру:

```asm
mov eax, [x] ; раз eax 32-битный, читаем dword
```

Но иногда размер нужно указать явно:

```asm
inc dword [x]
```

Потому что из `[x]` непонятно: увеличить байт, слово или двойное слово?

---

## Биты одинаковые, смысл разный

Регистр хранит биты. Он сам по себе не знает, signed это или unsigned.

```text
0xFFFFFFFF
```

Можно прочитать как:

```text
unsigned: 4294967295
signed:   -1
```

Это важно: типы C++ — это интерпретация. На уровне регистра лежит просто набор битов.

Позже это объяснит, почему есть разные переходы `jl` и `jb`, но нет отдельной команды `iadd`.

## Мини-челленджи

### 1. Разобрать `DEADBEEF`

Для:

```text
eax = 0xDEADBEEF
```

найди `ax`, `ah`, `al`.

<details>
<summary>Ответ</summary>

```text
ax = 0xBEEF
ah = 0xBE
al = 0xEF
```

</details>

### 2. Изменение `al`

Что будет с `eax`, если:

```text
eax = 0x11223344
```

и выполнить:

```asm
mov al, 0x99
```

<details>
<summary>Ответ</summary>

```text
eax = 0x11223399
```

Поменялся только младший байт.

</details>

### 3. Изменение `ah`

Что будет после:

```text
eax = 0xAABBCCDD
mov ah, 0x11
```

<details>
<summary>Ответ</summary>

```text
eax = 0xAABB11DD
```

`AH` — это байт `0xCC`, он заменился на `0x11`.

</details>

### 4. Какой размер?

Какой размер у `dword`?

<details>
<summary>Ответ</summary>

`dword` = 32 бита = 4 байта.

</details>

## Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| считать `eax` и `al` независимыми | `al` — часть `eax` |
| забыть, что `ah` тоже меняет `eax` | `ah` — байт внутри `ax` |
| путать `word` и `dword` | `word` = 16 бит, `dword` = 32 бит |
| думать, что регистр знает signed/unsigned | регистр хранит только биты |
| бездумно смешивать размеры | NASM может не собрать или получится не то |
