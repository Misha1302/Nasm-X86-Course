## Зачем этот день

В прошлом дне мы разобрали стек, `push`, `pop`, `call`, `ret`.

Теперь главный вопрос:

> Функция вызвалась. Как она понимает, где её аргументы, где локальные переменные и куда возвращать результат?

Ответ: через **стековый фрейм** и соглашение вызова **CDECL**.

Это одна из самых экзаменационных тем. Если ты понимаешь `[ebp+8]`, `[ebp-4]` и `add esp, 8`, половина ассемблерных листингов становится намного понятнее.

## Главная мысль

В классическом IA-32 CDECL:

```text
аргументы лежат выше ebp;
адрес возврата лежит по [ebp+4];
старый ebp лежит по [ebp];
локальные переменные лежат ниже ebp;
возвращаемое значение обычно в eax.
```

Карта:

```text
higher addresses
+------------------+
| argument 3       | [ebp+16]
+------------------+
| argument 2       | [ebp+12]
+------------------+
| argument 1       | [ebp+8]
+------------------+
| return address   | [ebp+4]
+------------------+
| old ebp          | [ebp]
+------------------+
| local variable 1 | [ebp-4]
+------------------+
| local variable 2 | [ebp-8]
+------------------+
lower addresses
```

## 1. Что такое фрейм функции

Фрейм функции — это кусок стека, который принадлежит конкретному вызову функции.

В нём могут быть:

- адрес возврата;
- старое значение `ebp`;
- аргументы;
- локальные переменные;
- временные значения;
- сохранённые регистры.

Если функция вызывается рекурсивно, каждый вызов получает свой фрейм.

```text
fact(5) frame
fact(4) frame
fact(3) frame
fact(2) frame
fact(1) frame
```

Именно поэтому локальные переменные рекурсивных вызовов не смешиваются.

## 2. Пролог функции

Классический пролог:

```asm
push ebp
mov ebp, esp
sub esp, N
```

Разберём.

### `push ebp`

Сохраняем старый `ebp`, чтобы потом вернуть фрейм вызывающей функции.

```text
esp -> [ old ebp ]
```

### `mov ebp, esp`

Теперь `ebp` становится стабильной базой текущего фрейма.

Почему нужен `ebp`, если есть `esp`?

Потому что `esp` меняется при `push`, `pop`, выделении временных данных. А `ebp` остаётся удобной точкой отсчёта.

### `sub esp, N`

Выделяем место под локальные переменные.

```asm
sub esp, 16
```

значит: зарезервировать 16 байт ниже `ebp`.

---

## 3. Эпилог функции

Классический эпилог:

```asm
mov esp, ebp
pop ebp
ret
```

Разберём.

### `mov esp, ebp`

Выкидываем локальные переменные: возвращаем `esp` к началу фрейма.

### `pop ebp`

Восстанавливаем старый `ebp` вызывающей функции.

### `ret`

Снимаем адрес возврата со стека и прыгаем обратно.

## 4. `leave`

Команда:

```asm
leave
```

примерно эквивалентна:

```asm
mov esp, ebp
pop ebp
```

Поэтому часто можно увидеть:

```asm
leave
ret
```

Это просто короткая форма эпилога.

## 5. Почему первый аргумент — `[ebp+8]`

После входа в функцию и пролога стек устроен так:

```text
[ebp+12]  second argument
[ebp+8]   first argument
[ebp+4]   return address
[ebp]     old ebp
[ebp-4]   local variable
```

Почему не `[ebp+4]`?

Потому что `[ebp+4]` уже занят адресом возврата, который положил `call`.

Почему не `[ebp]`?

Потому что `[ebp]` — сохранённый старый `ebp`.

Поэтому:

```text
first argument = [ebp+8]
```

Это нужно довести до автоматизма.

## 6. CDECL: правила вызова

CDECL — соглашение о том, как C-функции вызывают друг друга в 32-битном x86.

Главные правила:

| Правило | Что значит |
|---|---|
| arguments right-to-left | аргументы кладутся на стек справа налево |
| caller cleans stack | вызывающий после `call` делает `add esp, ...` |
| return value in `eax` | обычный `int` возвращается в `eax` |
| `edx:eax` for some 64-bit results | 64-битный integer может возвращаться через пару регистров |
| `eax/ecx/edx` caller-saved | функция может их испортить |
| `ebx/esi/edi` callee-saved | функция должна восстановить, если использовала |

## 7. Пример функции `sum`

C:

```c
int sum(int a, int b) {
    return a + b;
}
```

NASM:

```asm
sum:
    push ebp
    mov ebp, esp

    mov eax, [ebp+8]     ; a
    add eax, [ebp+12]    ; a + b

    pop ebp
    ret
```

Здесь:

```text
[ebp+8]  = a
[ebp+12] = b
eax      = return value
```

---

## 8. Вызов `sum`

C:

```c
int z = sum(a, b);
```

NASM:

```asm
push dword [b]
push dword [a]
call sum
add esp, 8
; eax = result
```

Почему сначала `b`, потом `a`?

Потому что CDECL кладёт аргументы справа налево.

Почему `add esp, 8`?

Два аргумента по 4 байта.

## 9. Локальные переменные

C:

```c
int f(int x) {
    int y;
    y = x + 1;
    return y * 2;
}
```

NASM-идея:

```asm
f:
    push ebp
    mov ebp, esp
    sub esp, 4          ; local y

    mov eax, [ebp+8]    ; x
    add eax, 1
    mov [ebp-4], eax    ; y = x + 1

    mov eax, [ebp-4]
    imul eax, 2

    mov esp, ebp
    pop ebp
    ret
```

Здесь:

```text
[ebp+8] = x
[ebp-4] = y
```

## 10. Caller-saved и callee-saved

Когда функция вызывается, нужно договориться, какие регистры кто сохраняет.

### Caller-saved

```text
eax, ecx, edx
```

Вызывающий не должен надеяться, что эти регистры сохранятся после вызова.

Если caller хочет сохранить их значения, он сам должен это сделать:

```asm
push eax
call f
pop eax
```

или сохранить в память.

### Callee-saved

```text
ebx, esi, edi
```

Если функция хочет использовать их, она должна восстановить перед возвратом.

```asm
my_func:
    push ebp
    mov ebp, esp
    push ebx

    ; use ebx

    pop ebx
    pop ebp
    ret
```

## 11. Почему `printf` портит регистры

Когда ты вызываешь:

```asm
call printf
```

после возврата нельзя рассчитывать, что `eax`, `ecx`, `edx` остались прежними.

Если перед `printf` в `eax` был важный ответ, а после `printf` ты снова хочешь его использовать, надо сохранить.

Например:

```asm
push eax
push eax
push fmtOut
call printf
add esp, 8
pop eax
```

Но часто проще: посчитать, напечатать и больше не использовать.

## 12. Рекурсия и фреймы

C:

```c
int fact(int n) {
    if (n <= 1) return 1;
    return n * fact(n - 1);
}
```

Каждый вызов имеет свой `[ebp+8]`, свой return address, свои локальные значения.

```text
fact(5): [ebp+8] = 5
fact(4): [ebp+8] = 4
fact(3): [ebp+8] = 3
...
```

Это возможно потому, что каждый `call` создаёт новый фрейм на стеке.

#### 2. Почему локальные переменные имеют отрицательные смещения?

#### 3. Напиши `square(x)`

C:

```c
int square(int x) { return x*x; }
```

<details>
<summary>Ответ</summary>

```asm
square:
    push ebp
    mov ebp, esp

    mov eax, [ebp+8]
    imul eax, eax

    pop ebp
    ret
```

</details>

#### 4. Почему после `call sum` нужно `add esp, 8`?

<details>
<summary>Ответ</summary>

Потому что caller положил два аргумента по 4 байта и по CDECL сам должен убрать их со стека.

</details>

#### 5. Какие регистры callee должен восстановить?

<details>
<summary>Ответ</summary>

В CDECL обычно `ebx`, `esi`, `edi`.

</details>

## 14. Типовые ошибки

| Ошибка | Почему плохо |
|---|---|
| считать первый аргумент `[ebp+4]` | там return address |
| забыть восстановить `ebp` | ломается фрейм вызывающей функции |
| использовать `ebx` и не восстановить | нарушается CDECL |
| забыть `add esp, ...` после вызова | caller не очистил аргументы |
| хранить важное значение в `eax` перед `call` | функция может испортить `eax` |
| перепутать локальные и аргументы | локальные ниже `ebp`, аргументы выше |

## Рекурсия — это обычный `call`

Рекурсивная функция не требует новой магии. Она просто вызывает сама себя.

Главный риск: значения, которые нужны после recursive call, нельзя оставлять только в `eax/ecx/edx`. Их надо сохранить во фрейме или в callee-saved регистрах.
